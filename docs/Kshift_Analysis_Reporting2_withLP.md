프로젝트 8202P1705의 ACT014:
- MUL_WGT = **1,658.37** (계수)
- DAY_CAPA = **200** (RES004 상한)
- **초과 배율: 8.29배**

이 ACT를 어떤 날짜에 배정하더라도 해당 날짜의 SCD002 제약이 위반됨.

## 4. 목적 함수 문제

### 4.1 현재 상태

```
Minimize
 obj:
Subject To
```

**목적 함수가 비어있음** — SCD002가 Hard 제약으로 수식화되어 있어, 용량 초과 시 해를 찾지 못하고 즉시 Infeasible 판정.

### 4.2 원래 설계 의도와의 괴리

SCM_CASE_CNST_MASTER에서 SCD002의 설명:
> "일일 생산능력 초과 물량에 대해서 **감점** 적용"

DB 정의는 **감점(Soft 제약)**인데, LP 모델에서는 **Hard 제약(≤)**으로 구현되어 있음.
이것이 근본적인 설계-구현 불일치.

만약 SCD002가 Soft 제약이었다면:
```
Minimize
 obj: 2 × slack_RES004_D20250711 + 2 × slack_RES004_D20250710 + ...

Subject To
 SCD002.RES004.D20250711: 1658.37 × x_ACT014_0711 + ... - slack_RES004_D20250711 <= 200
```

이 경우 용량 초과를 허용하되 패널티를 부과하므로, 해를 찾을 수 있음.

## 5. 해결 방안

### 5.1 방안 A: SCD002를 Soft 제약으로 전환 (권장, 설계 의도 복원)

**OptManager.cs의 SCD002 제약 생성 코드를 수정:**

현재 (Hard):
```csharp
IloCnstList.Add(AddLe(expr, dayCapacity, constraintName));
```

변경 (Soft with penalty):
```csharp
INumVar slack = NumVar(0, double.MaxValue, NumVarType.Float, "slack_" + constraintName);
IloCnstList.Add(AddLe(Sum(expr, Prod(-1, slack)), dayCapacity, constraintName));
objective.AddTerm(penaltyWeight, slack);  // penaltyWeight = SCD002.PENALTY_VAL (기본 2)
```

- 효과: 용량 초과를 허용하되 패널티 최소화 → 해를 찾으면서 용량 초과를 줄이는 최적 해 도출
- 소요: 1~2일 (OptManager.cs 수정)
- 위험도: 낮음 (DB 정의와 일치하는 원래 설계 의도 복원)

### 5.2 방안 B: DAY_CAPA 값 현실화

```sql
-- 현재 마킹팀(RES004) DAY_CAPA = 200
-- 실제 일일 처리 가능 물량이 2,000톤이라면:
UPDATE SCM_RESOURCE_MASTER
SET DAY_CAPA = 2000
WHERE CASE_NO = 'KSHIFT' AND COMPANY_NO = '1002' AND RESOURCE_NO = 'RES004';
```

- 효과: 대부분의 프로젝트가 용량 내에 들어감
- 소요: 30분 (DB UPDATE + 리소스 마스터 검토)
- 위험도: 중간 (실제 용량과 일치하는지 현장 확인 필요)

### 5.3 방안 C: MUL_WGT를 일일 물량으로 분할

현재: ACT 전체 물량(1,658.37)을 1일(STD_TRM=1)에 처리하는 것으로 모델링
개선: STD_TRM > 1로 설정하여 다일 스케줄링, 일일 물량 = MUL_WGT / STD_TRM

```
현재: 1658.37 × x_0711 (1일에 전체 물량)
개선: 184.26 × x_0703 + 184.26 × x_0704 + ... (9일에 분산, 1658.37 ÷ 9 = 184.26)
```

- 효과: 일일 물량이 DAY_CAPA 이내로 감소
- 소요: 코드 수정 필요 (변수 생성 방식 변경)
- 위험도: 높음 (모델 구조 변경)

### 5.4 방안 D: 대형 블록 제외 후 별도 처리

11개 초과 프로젝트를 자동 검출하여 최적화 대상에서 제외하고, 수동 배정 후 나머지만 최적화.

- 효과: 나머지 72개 프로젝트는 정상 최적화 가능
- 소요: 30분 (MakeOptActList 필터링 로직)
- 위험도: 낮음 (부분 해라도 확보)

## 7. 사전 진단 SQL (LP 생성 전 문제 감지)

```sql
-- 진단: MUL_WGT > DAY_CAPA인 ACT 검출 (LP 생성 전에 경고)
SELECT
    A.PROJECT_NO, A.ACT_NO, A.ACT_TYPE,
    D.RES_NO, R.RESOURCE_NAME,
    D.MUL_WGT, R.DAY_CAPA,
    ROUND(D.MUL_WGT / NULLIF(R.DAY_CAPA, 0), 2) AS 초과배율
FROM SCM_ACT_MASTER A
JOIN SCM_ACT_DETAIL D ON A.CASE_NO=D.CASE_NO AND A.COMPANY_NO=D.COMPANY_NO
     AND A.PROJECT_NO=D.PROJECT_NO AND A.ACT_NO=D.ACT_NO
JOIN SCM_RESOURCE_MASTER R ON D.CASE_NO=R.CASE_NO AND D.COMPANY_NO=R.COMPANY_NO
     AND D.RES_NO=R.RESOURCE_NO
JOIN SCM_CONTRACT_MASTER B ON A.CASE_NO=B.CASE_NO AND A.COMPANY_NO=B.COMPANY_NO
     AND A.PROJECT_NO=B.PROJECT_NO
WHERE A.CASE_NO = 'KSHIFT' AND A.COMPANY_NO = '1002'
AND NVL(B.MP_DT, B.MP_INIT_DT) BETWEEN TO_DATE('2025-06-01','YYYY-MM-DD')
     AND TO_DATE('2025-07-31','YYYY-MM-DD')
AND A.ACT_LEVEL = 2
AND D.MUL_WGT > R.DAY_CAPA
ORDER BY D.MUL_WGT / R.DAY_CAPA DESC;
```

## 9. 결론

LP 파일 분석을 통해 근본 원인이 **확정**되었습니다.

**문제**: SCD002(생산능력 준수)가 Hard 제약으로 구현되어 있으나, 대형 블록(MUL_WGT > 200)이 존재하여 어떤 날짜에도 배정 불가능.

**설계-구현 불일치**: DB에는 "감점 적용"으로 정의되어 있으나, 코드에서 Hard 제약(`AddLe`)으로 구현됨. 목적함수가 비어있는 것도 이 불일치의 증거.

**해결**: SCD002를 Soft 제약으로 전환하고 패널티를 목적함수에 반영하면, 용량 초과를 허용하면서 최소화하는 최적 해를 도출할 수 있음.
