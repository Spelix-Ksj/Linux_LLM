## 1. SSH 접속 방법

SSH는 원격 서버에 접속하기 위한 프로토콜입니다. Windows의 터미널(PowerShell, Windows Terminal, CMD)에서 실행합니다.

### 서버 접속

```bash
ssh root@192.168.10.40
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | Windows에서 Linux 서버(192.168.10.40)에 `root` 사용자로 원격 접속합니다. |
| **기대 결과** | 접속 성공 시 서버의 명령 프롬프트(`[root@서버명 ~]#`)가 표시됩니다. |
| **주의사항** | SSH 키 인증이 설정되어 있으므로 별도의 비밀번호 입력은 불필요합니다. `Connection refused` 오류가 뜨면 서버가 꺼져 있거나 네트워크 문제입니다. |

### 접속 종료

```bash
exit
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | 현재 SSH 세션을 종료하고 내 컴퓨터로 돌아옵니다. |
| **기대 결과** | Windows 프롬프트로 돌아갑니다. |
| **주의사항** | `exit` 대신 `Ctrl + D`를 눌러도 동일하게 종료됩니다. |

## 2. GPU 상태 확인

이 서버는 H100 GPU 5장으로 LLM 모델을 실행합니다. GPU 상태를 확인하면 모델이 정상 동작 중인지 파악할 수 있습니다.

### 현재 상태 확인

```bash
nvidia-smi
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | 모든 GPU의 사용량, 온도, 메모리 사용량, 실행 중인 프로세스를 한눈에 보여줍니다. |
| **기대 결과** | GPU 0~4까지 5장이 표시됩니다. GPU 0~3에는 vLLM(GPT-OSS-120B) 프로세스가, GPU 4에는 vLLM(Qwen3-Coder) 프로세스가 보입니다. |
| **주의사항** | 명령 실행 시점의 상태를 한 번만 보여줍니다. |

**출력 읽는 법**:

```
+-------------------------------------------+
| GPU  Name            Temp   Memory-Usage  |
|   0  NVIDIA H100 NVL  45C   xxxMiB/95830MiB |  <-- GPU 0 (gpt-oss-120b)
|   1  NVIDIA H100 NVL  43C   xxxMiB/95830MiB |  <-- GPU 1 (gpt-oss-120b)
|   2  NVIDIA H100 NVL  44C   xxxMiB/95830MiB |  <-- GPU 2 (gpt-oss-120b)
|   3  NVIDIA H100 NVL  42C   xxxMiB/95830MiB |  <-- GPU 3 (gpt-oss-120b)
|   4  NVIDIA H100 NVL  40C   xxxMiB/95830MiB |  <-- GPU 4 (Qwen3-Coder)
+-------------------------------------------+
```

| 항목 | 정상 범위 | 주의 기준 |
|------|-----------|-----------|
| **Temp (온도)** | 40~60도 | 80도 이상이면 성능 저하 위험 |
| **Memory-Usage** | 대부분 사용 중 | 모델이 메모리를 많이 사용하는 것이 정상 |
| **GPU-Util (사용률)** | 요청 시 상승 | 대기 중에는 낮고, 요청 처리 중에는 높아짐 |

### 실시간 모니터링

```bash
nvidia-smi -l 2
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | GPU 상태를 2초마다 자동으로 갱신하며 실시간으로 보여줍니다. |
| **기대 결과** | 화면이 2초마다 자동으로 갱신됩니다. |
| **주의사항** | 종료하려면 `Ctrl + C` 키를 누릅니다. `-l` 뒤의 숫자를 바꾸면 갱신 간격(초)을 조절할 수 있습니다. |

## 3. 서비스 관리

이 서버에는 3개의 주요 서비스가 있습니다.

| 서비스 이름 | 역할 | 포트 |
|-------------|------|------|
| `text2sql-ui` | Gradio 웹 UI (사용자 인터페이스) | 7860 |
| `vllm` | GPT-OSS-120B 모델 서빙 (GPU 0~3) | 8000 |
| `vllm-qwen3-coder` | Qwen3-Coder 모델 서빙 (GPU 4) | 8001 |

### 서비스 상태 확인 (status)

```bash
systemctl status text2sql-ui
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | `text2sql-ui` 서비스가 현재 실행 중인지, 언제 시작되었는지, 오류가 있는지 보여줍니다. |
| **기대 결과** | `Active: active (running)` 이면 정상입니다. |
| **주의사항** | `Active: failed` 또는 `Active: inactive`이면 문제가 있는 것입니다. |

```bash
systemctl status vllm
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | GPT-OSS-120B 모델 서빙 서비스(vLLM)의 상태를 확인합니다. |
| **기대 결과** | `Active: active (running)` 이면 정상입니다. |
| **주의사항** | 서비스 시작 후 모델 로딩에 2~5분이 소요됩니다. 로딩 중에는 `active (running)`이지만 API 응답이 안 될 수 있습니다. |

```bash
systemctl status vllm-qwen3-coder
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | Qwen3-Coder 모델 서빙 서비스의 상태를 확인합니다. |
| **기대 결과** | `Active: active (running)` 이면 정상입니다. |
| **주의사항** | `vllm` 서비스와 동일하게 모델 로딩에 시간이 소요됩니다. |

### 서비스 재시작 (restart)

```bash
systemctl restart text2sql-ui
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | 웹 UI 서비스를 중지했다가 다시 시작합니다. 설정 파일 변경이 반영됩니다. |
| **기대 결과** | 재시작 후 `Active: active (running)` 상태가 됩니다. |
| **주의사항** | 재시작 중(약 수 초) 웹 UI에 접속할 수 없습니다. |

```bash
systemctl restart vllm
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | GPT-OSS-120B vLLM 서비스를 재시작합니다. 모델을 다시 로딩합니다. |
| **기대 결과** | 서비스가 재시작되고, 2~5분 후 모델 로딩이 완료됩니다. |
| **주의사항** | 모델을 GPU에 다시 로딩하므로 **2~5분**이 소요됩니다. 이 시간 동안 SQL 생성이 불가능합니다. |

```bash
systemctl restart vllm-qwen3-coder
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | Qwen3-Coder vLLM 서비스를 재시작합니다. |
| **기대 결과** | 서비스가 재시작되고, 모델 로딩 후 정상 서빙됩니다. |
| **주의사항** | 모델 재로딩에 시간이 소요됩니다. |

### 서비스 중지 (stop) / 시작 (start)

```bash
systemctl stop text2sql-ui     # 웹 UI 중지
systemctl start text2sql-ui    # 웹 UI 시작
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | `stop`은 서비스를 완전히 중지하고, `start`는 중지된 서비스를 시작합니다. |
| **기대 결과** | `stop` 후에는 `inactive (dead)` 상태, `start` 후에는 `active (running)` 상태가 됩니다. |
| **주의사항** | 다른 서비스(`vllm`, `vllm-qwen3-coder`)도 같은 방식으로 `stop`/`start`할 수 있습니다. |

### 서비스 로그 확인 (journalctl)

```bash
journalctl -u text2sql-ui -n 50 --no-pager
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | `text2sql-ui` 서비스의 최근 로그 50줄을 화면에 출력합니다. |
| **기대 결과** | 서비스의 동작 로그가 표시됩니다. 오류가 있으면 `ERROR`, `FAIL` 등의 키워드가 보입니다. |
| **주의사항** | `-n 50`의 숫자를 바꾸면 출력 줄 수를 조절할 수 있습니다. `--no-pager`는 결과를 한 번에 전부 출력합니다. |

```bash
journalctl -u vllm -n 50 --no-pager
journalctl -u vllm-qwen3-coder -n 50 --no-pager
```

다른 서비스의 로그도 같은 방식으로 확인합니다. `-u` 뒤의 서비스 이름만 변경하면 됩니다.

### 서비스 실시간 로그 모니터링

```bash
journalctl -u text2sql-ui -f
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | 서비스의 로그를 실시간으로 보여줍니다. 새 로그가 생길 때마다 자동으로 화면에 추가됩니다. |
| **기대 결과** | 화면에 실시간 로그가 계속 추가됩니다. |
| **주의사항** | 종료하려면 `Ctrl + C` 키를 누릅니다. 문제 재현 시 로그를 동시에 보면서 원인을 파악할 때 유용합니다. |

## 4. Conda 환경 관리

이 서버에는 두 개의 Conda 환경이 있습니다.

| 환경 이름 | 용도 | 설명 |
|-----------|------|------|
| `text2sql` | Gradio 앱 | 웹 UI 관련 Python 패키지 (langchain, gradio, oracledb 등) |
| `py312_sr` | vLLM | 모델 서빙 관련 Python 패키지 (vllm, torch 등) |

### 환경 활성화

```bash
conda activate text2sql
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | Gradio 앱용 Python 환경(`text2sql`)을 활성화합니다. |
| **기대 결과** | 프롬프트 앞에 `(text2sql)`이 표시됩니다. |
| **주의사항** | 환경을 활성화하지 않으면 필요한 패키지를 찾지 못해 오류가 발생할 수 있습니다. |

```bash
conda activate py312_sr
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | vLLM용 Python 환경(`py312_sr`)을 활성화합니다. |
| **기대 결과** | 프롬프트 앞에 `(py312_sr)`이 표시됩니다. |
| **주의사항** | vLLM 관련 작업 시에만 이 환경을 사용합니다. |

### 설치된 패키지 확인

```bash
pip list
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | 현재 활성화된 Conda 환경에 설치된 모든 Python 패키지 목록을 보여줍니다. |
| **기대 결과** | 패키지 이름과 버전이 나열됩니다. |
| **주의사항** | 반드시 원하는 Conda 환경을 먼저 활성화한 뒤 실행합니다. 환경마다 설치된 패키지가 다릅니다. |

### 패키지 설치

```bash
pip install 패키지명
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | 현재 활성화된 Conda 환경에 새 Python 패키지를 설치합니다. |
| **기대 결과** | `Successfully installed 패키지명` 메시지가 출력됩니다. |
| **주의사항** | 올바른 Conda 환경이 활성화된 상태에서 설치합니다. 특정 버전을 설치하려면 `pip install 패키지명==버전` 형식을 사용합니다. |

**예시**:
```bash
conda activate text2sql           # 먼저 환경 활성화
pip install requests==2.31.0      # 특정 버전 설치
pip list | grep requests          # 설치 확인
```

## 5. 파일 수정

Linux 서버에서 파일을 편집할 때는 `nano` 텍스트 에디터를 사용합니다. Windows의 메모장과 비슷하지만 마우스 대신 키보드로 조작합니다.

### nano 에디터로 파일 편집

```bash
nano /root/text2sql/app/config.py
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | `nano` 텍스트 에디터로 지정한 파일을 엽니다. |
| **기대 결과** | 파일 내용이 편집 가능한 상태로 표시됩니다. |
| **주의사항** | 편집 후 반드시 저장(`Ctrl + O`)하고 종료(`Ctrl + X`)합니다. |

**nano 핵심 단축키**:

| 단축키 | 동작 |
|--------|------|
| `Ctrl + O` | 파일 저장. `Enter`를 눌러 확인합니다. |
| `Ctrl + X` | 에디터 종료. 저장하지 않았으면 저장 여부를 묻습니다. |
| `Ctrl + K` | 현재 줄 잘라내기 |
| `Ctrl + U` | 잘라낸 줄 붙여넣기 |
| `Ctrl + W` | 텍스트 검색 |
| 방향키 | 커서 이동 |

### 파일 내용 확인 (읽기 전용)

```bash
cat /root/text2sql/app/config.py
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | 파일의 전체 내용을 화면에 출력합니다. 수정 없이 내용만 확인합니다. |
| **기대 결과** | 파일 내용이 터미널에 출력됩니다. |
| **주의사항** | 파일이 길면 `less /root/text2sql/app/config.py`를 사용합니다. `q` 키로 종료합니다. |

### .env 비밀번호 변경

```bash
nano /root/text2sql/app/.env
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | `.env` 환경 변수 파일을 편집합니다. 웹 UI 비밀번호가 이 파일에 저장되어 있습니다. |
| **기대 결과** | 파일이 편집 모드로 열립니다. |
| **주의사항** | 변경 후 반드시 `systemctl restart text2sql-ui`를 실행해야 적용됩니다. |

**변경 순서**:

1. `nano /root/text2sql/app/.env` 실행합니다.
2. `GRADIO_PASSWORD=` 부분을 찾아서 `=` 뒤의 값을 새 비밀번호로 변경합니다.
   ```
   GRADIO_PASSWORD=새비밀번호
   ```
3. `Ctrl + O` -> `Enter` (저장)
4. `Ctrl + X` (에디터 종료)
5. 서비스 재시작:
   ```bash
   systemctl restart text2sql-ui
   ```

## 6. vLLM 모델 관리

vLLM은 대규모 언어 모델(LLM)을 빠르게 서빙하기 위한 엔진입니다.

### 현재 서빙 중인 모델 확인

```bash
curl http://localhost:8000/v1/models
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | GPT-OSS-120B 모델 서버(포트 8000)에 어떤 모델이 서빙되고 있는지 확인합니다. |
| **기대 결과** | JSON 형태로 모델 목록이 반환됩니다. |
| **주의사항** | `Connection refused` 오류가 나오면 vLLM 서비스가 시작되지 않았거나 모델 로딩 중입니다. 2~5분 후 다시 시도합니다. |

```bash
curl http://localhost:8001/v1/models
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | Qwen3-Coder 모델 서버(포트 8001)에 어떤 모델이 서빙되고 있는지 확인합니다. |
| **기대 결과** | JSON 형태로 모델 목록이 반환됩니다. |
| **주의사항** | 포트 번호가 8000이 아닌 **8001**입니다. |

### 모델 테스트 요청

```bash
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "/install_file_backup/tessinu/gpt-oss-120b",
    "messages": [{"role": "user", "content": "SELECT 1 FROM DUAL"}],
    "max_tokens": 50,
    "temperature": 0.0
  }'
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | GPT-OSS-120B 모델에 간단한 테스트 요청을 보냅니다. |
| **기대 결과** | JSON 형태의 응답이 반환됩니다. `"choices"` 배열에 모델 응답이 포함됩니다. |
| **주의사항** | `curl` 명령이 길어서 `\`(백슬래시)로 줄을 나누었습니다. 한 줄로 이어서 입력해도 됩니다. 응답이 올 때까지 수 초가 걸릴 수 있습니다. |

## 7. 방화벽 관리

방화벽은 서버로 들어오는 네트워크 트래픽을 제어합니다. 특정 포트를 열어야 외부에서 해당 서비스에 접근할 수 있습니다.

### 현재 열린 포트 확인

```bash
firewall-cmd --list-ports
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | 방화벽에서 현재 외부 접근을 허용하고 있는 포트 목록을 보여줍니다. |
| **기대 결과** | `7860/tcp 8000/tcp 8001/tcp` 등이 표시됩니다. |
| **주의사항** | 특정 서비스에 외부에서 접근이 안 될 때 해당 포트가 목록에 있는지 확인합니다. |

### 새 포트 열기

```bash
firewall-cmd --permanent --add-port=포트번호/tcp
firewall-cmd --reload
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | 첫 번째 명령으로 방화벽 규칙에 새 포트를 영구적으로 추가하고, 두 번째 명령으로 변경 사항을 적용합니다. |
| **기대 결과** | `success` 메시지가 출력됩니다. |
| **주의사항** | `--permanent` 옵션이 없으면 서버 재부팅 시 설정이 사라집니다. `--reload`를 해야 변경 사항이 즉시 적용됩니다. |

**예시: 포트 9090 열기**:
```bash
firewall-cmd --permanent --add-port=9090/tcp
firewall-cmd --reload
firewall-cmd --list-ports   # 추가되었는지 확인
```

## 8. 파일 업로드/다운로드

SCP는 SSH를 기반으로 파일을 안전하게 전송하는 명령어입니다. 아래 명령어들은 **Windows 터미널에서**(SSH 접속 전 상태에서) 실행합니다.

### Windows에서 서버로 파일 업로드

```bash
scp "D:/Dev/Linux_LLM/app/config.py" root@192.168.10.40:/root/text2sql/app/config.py
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | Windows PC의 `config.py` 파일을 Linux 서버로 복사합니다. |
| **기대 결과** | 파일 전송 진행률이 표시되고 완료됩니다. |
| **주의사항** | Windows 경로에 한글이나 공백이 있으면 반드시 큰따옴표(`"`)로 감쌉니다. 같은 이름의 파일이 서버에 있으면 덮어씌웁니다. |

### 서버에서 Windows로 파일 다운로드

```bash
scp root@192.168.10.40:/root/text2sql/app/config.py "D:/Dev/Linux_LLM/app/config.py"
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | Linux 서버의 `config.py` 파일을 Windows PC로 복사합니다. |
| **기대 결과** | 파일 전송 진행률이 표시되고 완료됩니다. |
| **주의사항** | 순서가 `scp 원본(서버) 대상(로컬)` 입니다. 업로드와 반대 순서입니다. |

### 폴더 전체 전송

```bash
# 폴더 업로드 (Windows -> Linux)
scp -r "D:/Dev/Linux_LLM/app" root@192.168.10.40:/root/text2sql/app

# 폴더 다운로드 (Linux -> Windows)
scp -r root@192.168.10.40:/root/text2sql/app "D:/Dev/Linux_LLM/app"
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | `-r` 옵션을 추가하면 폴더 전체(하위 파일 포함)를 전송합니다. |
| **기대 결과** | 폴더 내 모든 파일이 전송됩니다. |
| **주의사항** | 같은 이름의 파일이 있으면 덮어씌웁니다. 중요한 파일은 미리 백업합니다. |

## 9. 시스템 리소스 확인

서버의 전반적인 상태(메모리, 디스크, CPU)를 점검하는 명령어들입니다.

### 메모리(RAM) 사용량 확인

```bash
free -h
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | 서버의 전체 메모리, 사용 중인 메모리, 남은 메모리를 GB/MB 단위로 보여줍니다. |
| **기대 결과** | `total`, `used`, `free`, `available` 항목이 표시됩니다. |
| **주의사항** | `available` 항목이 실제로 사용 가능한 메모리입니다. 이 값이 2GB 미만이면 메모리 부족입니다. |

**출력 읽는 법**:
```
              total    used    free    shared  buff/cache   available
Mem:          503Gi   480Gi   1.2Gi    12Mi      22Gi        20Gi
```

### 디스크 사용량 확인

```bash
df -h
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | 서버의 디스크(저장공간) 사용량을 파티션별로 보여줍니다. |
| **기대 결과** | 각 파티션의 전체 크기, 사용량, 남은 공간, 사용률이 표시됩니다. |
| **주의사항** | `Use%`가 90% 이상이면 공간 확보가 필요합니다. `/` (루트 파티션)과 모델 저장 경로를 확인합니다. |

### CPU 사용률 확인

```bash
top
```

| 항목 | 설명 |
|------|------|
| **한글 설명** | CPU 사용률, 프로세스 목록, 메모리 사용량 등을 실시간으로 보여줍니다. |
| **기대 결과** | 실행 중인 프로세스 목록과 CPU/메모리 사용 현황이 실시간으로 표시됩니다. |
| **주의사항** | **`q` 키를 눌러야 종료됩니다.** `Ctrl + C`로도 종료할 수 있습니다. |

## 10. 부록: 자주 쓰는 Linux 기본 명령어

서버에서 자주 사용하는 기본 명령어 15개를 정리합니다.

| 번호 | 명령어 | 설명 | 예시 | 주의사항 |
|------|--------|------|------|----------|
| 1 | `ls` | 현재 폴더의 파일 목록 보기 | `ls /root/text2sql/` | |
| 2 | `ls -la` | 파일 목록 상세 보기 (숨김 파일 포함) | `ls -la /root/text2sql/` | `.`으로 시작하는 숨김 파일도 표시됨 |
| 3 | `cd` | 폴더 이동 | `cd /root/text2sql/app` | `cd ~`는 홈 디렉토리로 이동 |
| 4 | `pwd` | 현재 경로 확인 | `pwd` | 내가 어디 있는지 모를 때 사용 |
| 5 | `cat` | 파일 내용 출력 | `cat config.py` | 긴 파일은 `less` 사용 |
| 6 | `nano` | 파일 편집 | `nano config.py` | `Ctrl+O` 저장, `Ctrl+X` 종료 |
| 7 | `cp` | 파일 복사 | `cp config.py config.py.bak` | 백업 시 `.bak` 확장자 활용 |
| 8 | `mv` | 파일 이동 또는 이름 변경 | `mv old.py new.py` | |
| 9 | `rm` | 파일 삭제 | `rm 파일명` | **삭제 후 복구 불가** |
| 10 | `mkdir` | 새 폴더 만들기 | `mkdir /root/backup` | |
| 11 | `grep` | 파일에서 텍스트 검색 | `grep "error" logfile.txt` | `-i` 옵션으로 대소문자 무시 |
| 12 | `tail -f` | 파일 끝부분 실시간 모니터링 | `tail -f /var/log/messages` | `Ctrl+C`로 종료 |
| 13 | `history` | 이전에 실행한 명령어 기록 보기 | `history` | `!번호`로 재실행 가능 |
| 14 | `clear` | 터미널 화면 지우기 | `clear` | `Ctrl+L`도 동일 |
| 15 | `reboot` | 서버 재부팅 | `reboot` | **최후의 수단!** 모든 서비스가 중단됨 |

> **경고**: `rm -rf`와 `reboot`는 매우 신중하게 사용해야 합니다. 특히 `rm -rf /`는 절대로 실행하지 마세요. 서버의 모든 데이터가 삭제됩니다.
