## 1. 개요

전환배치계획 화면에서 자동배치(최적화) 수행 시 CPlex 솔버가 Root node에서 즉시 infeasible을 반환하는 문제를 분석하였다.

이전 1차 분석(HDTP-OPT-2026-003, FTR_MOVE_STD_ID=202408)에서는 **TEAM030 vs TEAM040 충돌(2개 제약)**이 원인이었으나, 이번 2차 분석(FTR_MOVE_STD_ID=202409)은 **TEAM001 vs TEAM040 충돌(15개 제약)**으로 성격이 근본적으로 다르다. 1차는 개인 속성 간의 논리적 충돌이었고, 2차는 조직 TO(Turnover) 자원의 구조적 부족 문제이다.

### 5.2 충돌 구조도

```
┌─────────────────────────────────────────────────────────┐
│                    수요 (TEAM040)                         │
│   9명의 직원이 각각 최소 1곳에 배치 필요 = 9슬롯 필요     │
│   E40, E94, E192, E365, E477, E478, E515, E561, E667    │
├─────────────────────────────────────────────────────────┤
│                    공급 (TEAM001)                         │
│   6개 조직의 가용 TO 합계 = 8슬롯 공급                    │
│   O12017(1) + O12036(1) + O12070(1)                      │
│   + O12139(2) + O12180(2) + O12195(1) = 8                │
├─────────────────────────────────────────────────────────┤
│              ★ 부족분: 9 - 8 = 1슬롯 ★                   │
│         → 수학적으로 해 불가능 (Infeasible)               │
└─────────────────────────────────────────────────────────┘
```

### 5.3 왜 이 6개 조직만 가능한가?

- 9명의 **PROP9(가능직무)**에 해당하는 JOB_TYPE2를 가진 레벨5 조직이 정확히 이 6개뿐이다.
- 이 6개 조직은 **다른 TEAM040 그룹과 공유되지 않으므로**, 외부 조정으로 해결이 불가능하다.
- 가용 TO(잔류원 제외)가 이미 최대한 반영된 상태이다.

## 6. TEAM001 제약 코드 상세

`OptManager.cs` 라인 327-348에서 TO 제약을 생성하는 코드는 다음과 같다.

```csharp
// TO 제약
foreach (var org in planManager.OrgItems)
{
    double limit = org.AlgTotTO - org.StayEmpCnt;
    string name = string.Format("TEAM001.O{0}", org.MoveOrg.ORG_ID);
    List<INumVar> varList = org.GetVars();
    if (varList != null && varList.Count > 0)
    {
        if (limit > 0)
            IloCnstList.Add(AddLe(Sum(varList.ToArray()), limit, name));
        else
            IloCnstList.Add(AddLe(Sum(varList.ToArray()), 0, name));
    }
}
```

### 7.2 방안 1 상세: TO 1 증가 (추천 즉시 조치)

6개 조직 중 TO를 증가시킬 최적 후보는 기존 TO가 큰 조직이 운영 부담이 적으므로, **O12139 (현재 TO=2)** 또는 **O12180 (현재 TO=2)**을 TO=3으로 변경하는 것을 권고한다.

**적용 SQL:**

```sql
-- 방안 1: MOVE_CASE_ORG 테이블에서 TO 변경
-- O12139의 TO를 2 → 3으로 변경 (예시)
UPDATE MOVE_CASE_ORG
SET TOT_TO = 3
WHERE FTR_MOVE_STD_ID = 202409 AND CASE_ID = 1
  AND CASE_DET_ID = '0' AND REV_ID = '000'
  AND ORG_ID = 12139;
COMMIT;

-- 또는 MOVE_ORG_MASTER에서 변경 (CaseOrg가 NULL인 경우)
UPDATE MOVE_ORG_MASTER
SET TOT_TO = 3
WHERE FTR_MOVE_STD_ID = 202409 AND ORG_ID = 12139;
COMMIT;
```

**롤백 SQL:**

```sql
UPDATE MOVE_CASE_ORG SET TOT_TO = 2
WHERE FTR_MOVE_STD_ID = 202409 AND CASE_ID = 1
  AND CASE_DET_ID = '0' AND REV_ID = '000' AND ORG_ID = 12139;
COMMIT;
```

### 7.3 방안 2 상세: 가능직무 확대

9명 중 1명의 PROP9에 다른 직무를 추가하여 6개 조직 외의 조직에도 배치 가능하게 한다. 인사팀에서 해당 직원의 역량 및 자격을 확인한 후 적합한 직무를 추가해야 한다.

### 7.4 방안 3 상세: 대상 제외

9명 중 1명을 MOVE_CASE_ITEM에서 이동 대상에서 제외(삭제 또는 FIXED_YN 설정)하여 수요를 8로 감소시키는 방안. 인사정책상 이동 대상 선정의 재검토가 필요하므로 인사팀 협의 필수.

### 7.5 방안 4 상세: TEAM040 소프트 전환

DB에서 TEAM040의 CNST_GBN을 "감점"으로 변경하는 것만으로는 불충분하다. 현재 코드(`OptManager.cs`:1301-1331)에서 TEAM040은 `IsPenaltyCnst` 분기가 없이 항상 `AddGe`(하드)를 사용하므로, **코드 수정이 필요**하다.

### 7.6 방안 5 상세: 전처리 검증(중기)

자동배치 실행 전 직무별 TEAM040 대상 인원수와 해당 직무 조직의 가용 TO 합계를 비교하여 수요>공급인 직무 그룹을 사전 감지하고 사용자에게 경고. TransferPlanManager.cs의 RunOptimization()에 ValidateTeam040Supply() 메서드 호출 추가.
