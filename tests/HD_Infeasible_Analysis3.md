## 1. 개요

본 보고서는 전환배치 자동배치 최적화 실행 시 발생하는 **Infeasible(해 불가능)** 문제에 대한 3차 분석 결과를 기술한다.

- **1차 분석** (HDTP-OPT-2026-003): TEAM030 vs TEAM040 충돌, IIS 2개 — 개인 속성(기혼여성+C지역) 하드 제약 간 충돌
- **2차 분석** (HDTP-OPT-2026-004): TEAM001 vs TEAM040 충돌, IIS 15개 — 레벨5(파트) 단위 TO 부족
- **3차 분석** (본 보고서): **TEAM026 vs TEAM040 충돌, IIS 21-24개** — 레벨4(팀) 단위 고년차 직원 균등배분 용량 병목

3차 분석에서 발견된 핵심 패턴은 **상위 조직 수준(레벨4)의 용량 병목**으로, 이전 분석들과는 다른 계층의 제약 충돌이다. 아울러, 향후 사용자에게 해결방안을 자동으로 제시하는 시스템 개선 방향을 추가로 검토하였다.

## 2. 문제 현상

자동배치 최적화 실행 시 CPLEX 솔버가 Root node에서 즉시 Infeasible을 반환하였다.

### CPLEX 로그 요약

```
MIP Presolve eliminated 13976 rows and 54628 columns.
Reduced MIP has 1507 rows, 3344 columns, and 17800 nonzeros.
Root relaxation: infeasible (1363 iterations)
Refine conflict on 15606 members → 24개 IIS 수렴
```

### 핵심 현상

| 항목 | 내용 |
|------|------|
| Infeasible 발생 시점 | Root node (LP Relaxation 단계) |
| 전체 제약 수 | 15,606개 |
| IIS 수렴 결과 | 24개 (25회 iteration) |
| 충돌 제약 유형 | TEAM026 (고년차 용량) vs TEAM040 (가능직무 배치) |

Root node에서 즉시 Infeasible이 발생하였으므로, 정수 조건과 무관하게 **연속 완화(LP Relaxation) 단계에서부터 해가 존재하지 않는** 구조적 문제임을 확인하였다.

### 4.3 충돌 구조도

```
┌──────────────────────────────────────────────────────────┐
│                    수요 (TEAM040)                          │
│                                                            │
│   14명의 고년차 직원 x 각 1곳 배치 = 14슬롯 필요           │
│                                                            │
│   E81,  E82,  E139, E140, E181, E261, E262,               │
│   E265, E336, E337, E410, E414, E487, E706                │
├──────────────────────────────────────────────────────────┤
│                    공급 (TEAM026)                          │
│                                                            │
│   7개 상위 조직의 고년차 용량 합계 = 13슬롯               │
│                                                            │
│   O11009(3) + O11017(3) + O11022(1) + O11033(2)           │
│   + O11045(2) + O11054(1) + O11064(1) = 13                │
├──────────────────────────────────────────────────────────┤
│                                                            │
│             ★ 부족분: 14 - 13 = 1슬롯 ★                  │
│                                                            │
│        Hall's Marriage Theorem 위반                        │
│        → 수학적으로 해가 불가능함                          │
│                                                            │
└──────────────────────────────────────────────────────────┘
```

**Hall's Marriage Theorem 적용**: 14명의 직원을 7개 조직(총 용량 13)에 배치하는 문제에서, 수요(14) > 공급(13)이므로 어떠한 배치 조합으로도 모든 직원을 배치할 수 없다. 이는 단순 산술 부족을 넘어, 이분 그래프 매칭의 수학적 불가능성을 의미한다.

### 4.4 CPLEX IIS 24개 vs 검증 IIS 21개

| 구분 | IIS 크기 | 설명 |
|------|---------|------|
| CPLEX 보고 | 24개 | CPLEX가 내부 알고리즘으로 찾은 IIS |
| LP 검증 | 21개 | TEAM040 14개 + TEAM026 7개 |

- CPLEX가 찾은 24개는 유효한 IIS이나, **IIS(Irreducible Infeasible Subsystem)는 유일하지 않다**.
- LP 분석을 통해 검증한 최소 IIS는 21개로, TEAM040 14개 + TEAM026 7개로 구성된다.
- CPLEX 24개에서 추가된 3개는 일부 TEAM026 상위 제약이 TEAM001 하위 제약 + 추가 TEAM040으로 분해된 변형이다.
- 본질적인 충돌 구조는 동일하다.

## 5. TEAM026 제약 코드 분석

TEAM026 제약은 OptManager.cs의 팀 제약 섹션에서 구현되어 있다.

| 항목 | 내용 |
|------|------|
| 호출 위치 | `ApplyTeamConstraint()` 메서드 내부 |
| 적용 단위 | 레벨4 (팀) 단위 |
| 제약 형태 | `AddLe(Sum(고년차 변수), 용량상한)` |
| 제약 종류 | 하드 제약 (Less-than-or-Equal) |

구조적으로 해당 제약은 다음과 같이 동작한다:

1. 레벨4 팀 조직별로 고년차 직원에 해당하는 배치 변수를 수집
2. 해당 변수의 합이 지정된 용량 상한 이하가 되도록 제약 추가
3. 하드 제약이므로 위반 시 솔루션이 존재할 수 없음

### 6.2 방안 1 상세: TEAM026 용량 1 증가 (★ 추천 즉시 조치)

TEAM026 용량을 DB에서 변경한다. 운영 부담을 최소화하기 위해 **기존 용량이 가장 큰 조직**의 상한을 1 증가시키는 것을 권장한다.

**추천 대상**: O11009 (현재 3 → 4) 또는 O11017 (현재 3 → 4)

#### 적용 SQL

```sql
-- TEAM026 용량 변경 (O11009 기준)
-- TEAM026 용량은 MOVE_CASE_CNST_MASTER의 CNST_VAL 또는 VAL01 필드에 저장
-- 정확한 필드명은 TEAM026 코드에서 확인 필요
UPDATE MOVE_CASE_CNST_MASTER
SET    CNST_VAL = 4  -- 또는 VAL01 = 4
WHERE  FTR_MOVE_STD_ID = 202501
  AND  CASE_ID = 1
  AND  CASE_DET_ID = '0'
  AND  REV_ID = '000'
  AND  ORG_ID = 11009
  AND  CNST_CD = 'TEAM026';
COMMIT;
```

#### 롤백 SQL

```sql
-- 원상 복구
UPDATE MOVE_CASE_CNST_MASTER
SET    CNST_VAL = 3
WHERE  FTR_MOVE_STD_ID = 202501
  AND  CASE_ID = 1
  AND  CASE_DET_ID = '0'
  AND  REV_ID = '000'
  AND  ORG_ID = 11009
  AND  CNST_CD = 'TEAM026';
COMMIT;
```

> **참고**: 용량 변경 후 고년차 용량 합계가 14 이상이 되므로 수요(14)를 충족할 수 있다. 단, 직원별 배치 가능 조직의 분포에 따라 매칭이 불가능한 경우가 존재할 수 있으므로, 변경 후 반드시 재실행하여 Feasible 여부를 확인해야 한다.

### 6.3 방안 2 상세: TEAM026을 소프트 제약으로 전환

TEAM026을 하드 제약에서 소프트 제약(감점)으로 전환하여, 용량 초과 시 페널티만 부과하고 해가 존재하도록 허용한다.

#### 적용 SQL

```sql
UPDATE MOVE_CASE_CNST_MASTER
SET    CNST_GBN = '감점',
       PENALTY_VAL = 10
WHERE  FTR_MOVE_STD_ID = 202501
  AND  CASE_ID = 1
  AND  CASE_DET_ID = '0'
  AND  REV_ID = '000'
  AND  CNST_CD = 'TEAM026';
COMMIT;
```

> **주의사항**: TEAM026 코드가 `IsPenaltyCnst` 분기를 지원하는지 사전에 코드 확인이 필요하다. 지원하지 않을 경우 코드 수정이 선행되어야 한다.

## 9. 향후 사용자 해결방안 자동 제시 시스템

### 9.1 현재 시스템의 사용자 경험

현재 Infeasible 발생 시 시스템이 제공하는 정보는 다음과 같다.

| 단계 | 현재 동작 |
|------|----------|
| 1 | 경고 메시지: "해가 존재하지 않습니다. 제약위배사항을 확인하세요." |
| 2 | 제약위배정보 Grid에 충돌 제약 목록 표시 (제약코드, 제약명, 조직정보, 직원정보) |
| 3 | TEAM001 위배 시에만 "TO 위배" 통계 요약 제공 |

**현재 시스템의 한계:**

- "어떻게 해결해야 하는지"에 대한 안내가 없음
- 사용자가 직접 IIS를 해석하고 조치를 판단해야 함
- 동일 패턴이 반복되어도 매번 수작업 분석이 필요
- TEAM001 이외의 제약(TEAM026, TEAM030 등) 위배 시 요약 정보 부재

### 9.2 권장 개선 방향 (5단계)

#### Phase 1 — 즉시 (1주): ConflictItem에 권장 조치 추가

- `ConflictItem` 클래스에 `RecommendedAction` 프로퍼티 추가
- `MakeConflictInfo()` 실행 이후 제약 코드별 권장 조치 자동 생성
- Grid에 "권장 조치" 컬럼 추가

**제약별 권장 메시지 예시:**

| 충돌 패턴 | 권장 메시지 |
|-----------|------------|
| TEAM001 vs TEAM040 | "해당 직무 조직의 TO가 N명 부족합니다. TO를 N 증가하거나 대상 인원을 조정하세요." |
| TEAM026 vs TEAM040 | "팀 '{팀명}'의 고년차 배치 용량이 N명 부족합니다. TEAM026 용량을 증가하세요." |
| TEAM030 vs TEAM040 | "직원 '{성명}'은 기혼여성으로 C지역 배치가 금지되어 있으나 가능직무가 C지역에만 있습니다. TEAM030을 감점으로 변경하세요." |

#### Phase 2 — 2주: IIS 자동 요약 ("수요 vs 공급" 형태)

- IIS 구성을 파싱하여 "N명 → M개 조직(용량 K) → 부족 N-K" 형태로 요약
- `GetConflictSummaryInfo()` 메서드 확장
- 사용자가 IIS 세부 내용을 이해하지 못하더라도 수급 불일치를 직관적으로 파악 가능

#### Phase 3 — 1개월: 제약 자동 완화 옵션

- UI에 **"이 제약을 감점으로 완화하고 재최적화"** 버튼 추가
- 사용자가 충돌 제약을 선택하면 자동으로 `CNST_GBN`을 변경하고 재실행
- 재실행 후 원상 복구 옵션 제공
- 변경 이력 로깅

#### Phase 4 — 2개월: 전처리 수급 검증

- 자동배치 실행 전 TEAM040 대상 인원과 해당 조직 TO/용량을 교차 검증
- TEAM001, TEAM026, TEAM030 각각에 대해 수급 불일치 사전 감지
- 감지 시 사용자에게 경고 팝업 + 자동 조정 제안
- Hall's Marriage Theorem 기반 사전 매칭 가능성 검증

#### Phase 5 — 3개월: AI 기반 해결방안 제시

- 과거 분석 사례(1차/2차/3차)를 학습 데이터로 활용
- 유사 IIS 패턴 감지 시 기존 해결방안을 RAG(Retrieval-Augmented Generation)로 참조
- Local LLM 또는 Claude API를 활용한 자연어 해결방안 생성
- 사용자 피드백 루프를 통한 지속적 개선

## 10. 결론

본 건은 **TEAM026(고년차 직원 균등 배분) 상위 조직 용량(13슬롯)이 TEAM040(가능직무) 대상 인원(14명)보다 1 부족**하여 발생한 구조적 Infeasible이다.

### 3회 분석을 통해 확인된 Infeasible 발생 패턴 유형

| 유형 | 차수 | 충돌 | 해결 |
|------|------|------|------|
| **개인 속성 충돌** | 1차 | TEAM030(속성) vs TEAM040(직무) | 속성 제약 소프트 전환 |
| **파트 TO 부족** | 2차 | TEAM001(TO) vs TEAM040(직무) | TO 증가 |
| **팀 용량 부족** | 3차 | TEAM026(고년차) vs TEAM040(직무) | 용량 증가 또는 소프트 전환 |

### 핵심 발견

- **공통 패턴**: TEAM040(가능직무 하드 제약)이 다른 하드 제약과 충돌할 때 Infeasible이 발생한다.
- **근본 해결**: TEAM040 대상자의 가능직무 조직에 충분한 여유가 있는지 사전 검증하는 **전처리 시스템 구축**이 필요하다.
- **즉시 조치**: 본 건은 TEAM026 용량을 1 증가(O11009: 3→4 또는 O11017: 3→4)하여 해결 가능하다.
