# HDTP Infeasible 분석 방법론 (종합)

## 개요

HDTP 전환배치 최적화 시스템에서 CPLEX Infeasible 발생 시 분석하는 종합 방법론이다.
TEAM001~TEAM048 전체 제약을 대상으로 하며, 어떤 LLM이든 이 문서를 참조하면 동일한 분석을 수행할 수 있다.

## 1. 제약 분류 체계

### 1.1 제약 유형별 분류

| 등급 | 코드 범위 | 성격 | Infeasible 위험도 | 사용자 변경 |
|------|-----------|------|-------------------|------------|
| 절대 제약 (CnstType=1) | TEAM001~010 | 시스템 기본, TO/1인1조직 등 | **높음** (TEAM001) | 불가 |

## 2. 10단계 분석 절차

### Step 1: CPLEX 로그 파싱

**목표**: Infeasible 발생 시점과 규모 파악

1. CPLEX 로그에서 `infeasible` 키워드 검색
2. 발생 시점: Root node(LP Relaxation) vs Branch&Cut 도중
3. `Refine conflict on N members` → 전체 제약 수 N 기록
4. IIS 수렴: `Max Members` / `Min Members` 값에서 IIS 크기 확인

**해석 기준**:
- Root node infeasible → 연속 완화에서부터 불가 → 구조적 문제
- IIS 2~5개 → 개인 속성 충돌, IIS 10~20개 → 조직 TO 충돌, IIS 20+개 → 정책 분배 충돌

### Step 2: IIS (Irreducible Infeasible Subsystem) 식별
 - Irreducible Infeasible Subsystem (IIS, 축소 불가능한 불능 하위 시스템)은 선형 계획법(Linear Programming)이나 제약 조건 만족 문제에서 
   해(Solution)가 존재하지 않는(불능, Infeasible) 시스템의 최소 단위 제약 조건 집합을 의미

1. `MakeConflictInfo()` 출력 또는 LP 파일에서 IIS 제약명 추출
2. 제약명 파싱: `{CNST_CD}.E{EMP_ID}` 또는 `{CNST_CD}.O{ORG_ID}` 형식

### Step 3: IIS 제약 분류 — 수요 vs 공급

1. 수요 측: `AddGe(Sum, 1)` 형태 → "반드시 배치" (보통 TEAM040)
2. 공급 측: `AddLe(Sum, limit)` 형태 → "배치 상한" (TEAM001/020/026/030 등)
3. **TEAM040이 없는 IIS**의 경우 → 다른 제약 쌍의 직접 충돌 (아래 충돌 매트릭스 참조)

### Step 4: 관련 직원 데이터 조회

- PROP9(가능직무), 성별/혼인/자녀, PosLongevity(고년차), ChangeJobType2 등
- 각 직원의 배치 가능 조직 목록 확인

### Step 5: 관련 조직 데이터 조회

- AlgTotTO, StayEmpCnt → 가용 TO
- REGION_TYPE(A/B/C/D), JOB_TYPE2, LVL4 상위 조직
- TEAM026 용량 산출: TotTo>=7→4, >=5→2, >=3→1, <3→0 (StayLongevity 차감)

### Step 6: LP 파일 실증 검증

LP 파일(`emp_move_*.lp`)에서 IIS 제약명 검색, 변수 목록과 RHS 확인, 충돌 변수 교집합 식별

### Step 7: 수학적 불가능성 증명

상황에 맞는 방법 선택 (아래 수학적 증명 섹션 참조)

### Step 8: 근본 원인 종합

기술적 원인(제약 충돌) + 데이터 원인(직원/조직) + 설계 원인(Hard/Soft 적정성) + 이전 분석 비교

### Step 9: 해결방안 도출

카테고리 A(DB CNST_GBN 변경), B(데이터 조정), C(코드 변경), D(구조 개선) 중 선택

### Step 10: 해결방안 비교 평가

| 평가 항목 | 척도 |
|-----------|------|
| 리스크 | 낮음/중간/높음 |
| 복잡도 | 매우 낮음~매우 높음 |
| 영향 범위 | 개인/조직/전체 |
| 소요 시간 | 즉시/1주/1개월+ |
| 가역성 | 즉시 롤백/복잡/불가역 |
| 설계 부합성 | 부합/중립/상충 |

## 4. 제약별 분석 체크리스트

### TEAM001 (TO 준수) — 항상 Hard, Soft 분기 없음

- [ ] IIS 내 TEAM001.O{ID} 조직별 `AlgTotTO - StayEmpCnt` 확인
- [ ] **글로벌 부족** vs **분절 부족(fragmentation)** 구분:
  - 글로벌: 전체 직무 TO 합계 < 대상자 수 → TO 증가 필수
  - 분절: 전체 여유는 있으나 특정 소그룹에서 병목 → 가능직무 확대 또는 특정 조직 TO 조정
- [ ] 2차 분석 사례: 9명 vs 6조직(TO=8), 부족분 1슬롯, 분절 부족

### TEAM020 (사업소 인원 상하한) — Hard/Soft 분기 있음

- [ ] 상한 위반: 이동 인원이 사업소 상한 초과
- [ ] 하한 위반: 이동 후 잔류 인원이 사업소 하한 미달
- [ ] 양쪽 동시 위반 가능 → 상한/하한 값의 적정성 검토

### TEAM021 (남성 1인 이상) — Hard/Soft 분기 있음

- [ ] 팀/파트 내 남성 직원 0명 상태에서 남성 배치 요구
- [ ] 가용 남성 직원이 모두 다른 하드 제약에 의해 해당 조직 배치 불가 시 충돌

### TEAM024 (고졸 균등배분) — Hard/Soft 분기 있음

- [ ] TO 기반 상한이 고졸 대상자 수보다 작은 팀 존재 여부
- [ ] 고졸이면서 TEAM040 대상인 직원의 가능직무 조직 분포 확인

### TEAM026 (고년차 균등배분) — Hard/Soft 분기 있음

- [ ] LVL4 단위 용량 산출: TotTo>=7→4, >=5→2, >=3→1, <3→0 (StayLongevity 차감)
- [ ] 3차 분석 사례: 14명 vs 7조직(용량=13), 부족분 1슬롯
- [ ] Hall's Marriage Theorem 적용 가능 (이분 그래프 매칭)

### TEAM027/028 (직급연차 균등) — Hard/Soft 분기 있음

- [ ] 선임 4년/5년 분배 상하한 확인
- [ ] 파트 TO 2명 이상일 때만 적용 → 소규모 파트에서 충돌 가능성

### TEAM030 (워킹맘 C지역) — Hard/Soft 분기 있음

- [ ] 1차 분석 사례: E784, 가능직무 조직이 C지역 O12235 단 1개
- [ ] **CNST_GBN 설정 오류 패턴**: 코드 주석은 "배치지양"이나 DB는 "제약"(Hard)
- [ ] PENALTY_VAL=10 이미 설정 → "감점" 전환 시 즉시 작동

### TEAM033/035 (이동기간 제한) — Hard/Soft 분기 있음

- [ ] TEAM033: 전환배치 18개월 미만 이동 제한
- [ ] TEAM035: 24개월 이내 이동 제한
- [ ] 기간 미달 직원이 TEAM040 대상이고 가능직무 조직이 제한될 때 충돌

### TEAM037 (직무이동 설정) — Hard/Soft 분기 있음

- [ ] `ChangeJobType2 == false`인 직원의 직무변경 금지
- [ ] 직무변경 미설정자가 TEAM040에 의해 다른 직무 조직에 배치 강제될 때 충돌

### TEAM039 (신입 C지역 금지) — Hard/Soft 분기 있음

- [ ] A/B지역 24개월 이하 직원 → C지역 배치 금지
- [ ] 해당 직원의 가능직무 조직이 모두 C지역일 때 TEAM040과 충돌

### TEAM040 (가능직무) — **항상 Hard, 코드에 Soft 분기 없음**

- [ ] `Sum(가능직무 조직 변수) >= 1` — 최소 1개 배치 강제

- [ ] **모든 기존 Infeasible 사례(1~3차)에서 수요 측으로 관여**

### TEAM043/044 (승진대상자) — Hard/Soft 분기 있음

- [ ] 팀 단위 승진대상자 분배 상한

## 6. 수학적 증명 방법

### 방법 A: 직접 산술 증명 — 단순 속성 충돌 (IIS 2~5)

**사용 시점**: 1명의 직원에 대해 2개 제약이 직접 모순
**1차 분석 예제**: 직원 E784
- TEAM030: `Sum(C지역 변수) <= 0` → O12235 배치 금지
- TEAM040: `E784.O12235 >= 1` → O12235 배치 강제
- O12235는 C지역 → `0 <= -1` 모순, QED

### 방법 B: Pigeonhole Principle — 수급 부족 (IIS 10~20)

**사용 시점**: N명을 M슬롯에 배치해야 하는데 N > M
**2차 분석 예제**: 9명 vs 6조직(TO합=8)
- 9명 각각 최소 1곳 배치 필요 = 9슬롯 수요
- 6조직 가용 TO 합계 = 8슬롯 공급
- 9 > 8 → 비둘기집 원리에 의해 불가, QED

### 방법 C: Hall's Marriage Theorem — 이분 그래프 매칭 (IIS 20+)

**사용 시점**: 직원-조직 이분 그래프에서 용량 제한이 복잡할 때
**3차 분석 예제 (worked example)**:
- 14명의 고년차 직원, 7개 상위 조직 (용량 합계 13)
- 직원 집합 S = {E81, ..., E706} (|S|=14)
- 이웃 집합 N(S) = 7개 조직, 총 용량 = 13
- Hall's 조건: |S| > 용량(N(S)) → 14 > 13
- complete matching 불가, QED

### 방법 D: Farkas Lemma — 고급 LP 이론

**사용 시점**: 일반적으로 불필요. 위 3가지로 커버되지 않는 복잡한 경우에만 사용

### 방법 선택 기준

| IIS 크기 | 충돌 유형 | 추천 방법 |
|----------|-----------|-----------|
| 2~3 | 1인 속성 모순 | **방법 A** (직접 산술) |
| 5~20 | N명 vs M슬롯 단순 부족 | **방법 B** (Pigeonhole) |
| 20+ | 복잡 용량 분포, 매칭 문제 | **방법 C** (Hall's Theorem) |
| 특수 | 위 3가지 불충분 시 | **방법 D** (Farkas) |

### 8.2 섹션 구조 (9개 섹션)

1. **개요** — 분석 배경, 이전 차수와의 비교

3. **데이터 현황** — 대상자, 조직, TO, 수급 분석
4. **근본 원인 — IIS 상세 분석** — IIS 구성, 충돌 구조도, 수학적 증명, N차 비교표
5. **충돌 제약 코드 분석** — 관련 제약 메서드 상세
6. **해결방안** — 총괄 비교표 + 각 방안 상세
7. **영향 범위** — 직접/간접 영향
8. **검증 방법** — 적용 후 확인 절차
9. **결론** — 패턴 요약, 즉시/중기 권고

- 충돌 구조도: ASCII 박스 다이어그램 (`┌─┐ │ └─┘`)
- 수급 표현: `N명 수요 > M슬롯 공급 → 부족분 K`
- SQL: 항상 적용 SQL + 롤백 SQL 쌍
- 모든 수치에 출처(LP 파일, DB, 코드) 명시
- 한국어 용어: 하드 제약, 감점, 제약위배, 자동배치, 전환배치
