# HDTP Infeasible 분석 방법론

## 개요

이 문서는 HDTP 전환배치 최적화 시스템에서 CPLEX Infeasible 발생 시 분석하는 10단계 방법론이다.
어떤 LLM이든 이 문서를 참조하면 동일한 분석을 수행할 수 있다.

---

## 1. 10단계 분석 절차

### Step 1: CPLEX 로그 파싱

**목표**: Infeasible 발생 시점과 규모 파악

**행동**:
1. CPLEX 로그에서 `infeasible` 키워드 검색
2. 발생 시점 확인: Root node(LP Relaxation) vs Branch&Cut 도중
3. `Refine conflict on N members` → 전체 제약 수 N 기록
4. IIS 수렴 결과: `Max Members` / `Min Members` 값에서 IIS 크기 확인

**산출물**: Infeasible 발생 시점, 전체 제약 수, IIS 크기

**해석 기준**:
- Root node infeasible → 연속 완화에서부터 불가 → 구조적 문제
- Branch&Cut 중 infeasible → 정수 조건과 결합된 문제
- IIS 2-3개 → 단순 충돌 (개인 속성)
- IIS 10개 이상 → 구조적 수급 불일치

### Step 2: IIS (Irreducible Infeasible Subsystem) 식별

**목표**: 충돌 제약 목록 확보

**행동**:
1. `MakeConflictInfo()` 메서드의 출력 확인 (ConflictItem 목록)
2. 또는 LP 파일에서 IIS에 해당하는 제약명 추출
3. 제약명 파싱: `{CNST_CD}.E{EMP_ID}` 또는 `{CNST_CD}.O{ORG_ID}` 형식

**산출물**: IIS 제약 목록 (제약코드, 직원ID, 조직ID)

### Step 3: IIS 제약 분류

**목표**: 수요 측과 공급 측 분리

**행동**:
1. IIS 내 제약을 제약코드별로 그룹핑
2. 수요 측 식별 (보통 TEAM040 — "반드시 배치"):
   - `AddGe(Sum, 1)` 형태 → 최소 1개 배치 요구
3. 공급 측 식별 (TEAM001, TEAM026, TEAM030 등 — "배치 제한"):
   - `AddLe(Sum, limit)` 형태 → 배치 상한 제한

**산출물**: 수요 제약 N개 + 공급 제약 M개

### Step 4: 관련 직원 데이터 조회

**목표**: 충돌 직원의 속성 확인

**행동**:
1. IIS에 포함된 직원 ID 목록 추출
2. 각 직원의 속성 확인:
   - PROP9 (가능직무) → TEAM040 배치 가능 조직
   - 성별/혼인/자녀 → TEAM030 해당 여부
   - PosLongevity (고년차) → TEAM026 해당 여부
   - 기타 속성
3. 각 직원의 배치 가능 조직 목록 확인

**산출물**: 직원별 속성 및 배치 가능 조직 테이블

### Step 5: 관련 조직 데이터 조회

**목표**: 공급 측 용량 확인

**행동**:
1. IIS에 포함된 조직 ID 목록 추출
2. 각 조직의 속성 확인:
   - AlgTotTO, StayEmpCnt → 가용 TO
   - REGION_TYPE → 지역 분류 (A/B/C/D)
   - JOB_TYPE2 → 직무 분류
   - LVL4 상위 조직 → TEAM026 등 상위 제약 대상
3. 용량 상한 계산

**산출물**: 조직별 용량 및 속성 테이블

### Step 6: LP 파일 실증 검증

**목표**: 수학식 레벨에서 충돌 확인

**행동**:
1. LP 파일(`emp_move_*.lp`)에서 IIS 제약명 검색
2. 각 제약의 정확한 변수 목록과 RHS 값 확인
3. 수요 측 변수와 공급 측 변수의 교집합 확인
4. 충돌 지점 (동일 변수가 양쪽에 모두 등장) 식별

**산출물**: 제약별 수학식 및 충돌 변수 목록

### Step 7: 수학적 불가능성 증명

**목표**: Infeasible이 수학적으로 불가피함을 증명

**증명 방법** (아래 4가지 중 상황에 맞는 것 선택):

#### 방법 A: 직접 산술 증명 (단순 충돌)
- 1차 분석 사례: TEAM030이 O12235 배치를 금지 + TEAM040이 O12235에 반드시 배치 요구
- "0 <= -1" 형태의 직접 모순

#### 방법 B: Pigeonhole Principle (수급 부족)
- 2차, 3차 분석 사례: N명을 M슬롯에 배치, N > M
- "9명 > 8슬롯" → 비둘기집 원리에 의해 불가

#### 방법 C: Hall's Marriage Theorem (이분 그래프 매칭)
- 직원-조직 이분 그래프에서 complete matching 존재 불가 증명
- 직원 집합 S에 대해 |N(S)| < |S| (이웃 집합의 크기가 집합 크기보다 작음)

#### 방법 D: 선형 프로그래밍 Farkas Lemma
- 고급 증명: 원문제의 dual이 unbounded임을 보임
- 일반적으로 필요하지 않음

**산출물**: 수학적 증명 (1~2문단)

### Step 8: 근본 원인 종합

**목표**: 비즈니스 관점에서 원인 요약

**행동**:
1. 기술적 원인 (어떤 제약이 어떻게 충돌하는지)
2. 데이터 원인 (어떤 직원/조직 데이터가 충돌을 야기하는지)
3. 설계 원인 (하드/소프트 설정이 적절한지, 원래 설계 의도와 일치하는지)
4. 이전 분석과의 비교 (패턴 유사성)

**산출물**: 원인 요약 테이블 (순서별 인과관계)

### Step 9: 해결방안 도출

**목표**: 복수의 해결방안 제시 및 비교

**해결방안 카테고리 4가지**:

#### 카테고리 A: DB 변경 (CNST_GBN 변경)
- 충돌 제약의 CNST_GBN을 "제약"→"감점"으로 변경
- 적용 조건: 해당 제약 코드가 `IsPenaltyCnst` 분기를 지원해야 함
- **주의**: TEAM040은 코드에 분기가 없으므로 DB만 변경해도 효과 없음

#### 카테고리 B: 데이터 조정
- TO 증가, 가능직무 확대, 대상 제외 등
- 직접적이지만 개별적 해결 (확장성 제한)

#### 카테고리 C: 코드 변경
- 제약 메서드에 `IsPenaltyCnst` 분기 추가
- 전처리 검증 로직 추가
- 확장적이지만 개발/배포 필요

#### 카테고리 D: 구조 개선
- 우선순위 기반 자동 완화 체계
- AI 기반 해결방안 자동 제시
- 장기적 과제

**산출물**: 방안별 비교표

### Step 10: 해결방안 비교 평가

**평가 매트릭스**:

| 평가 항목 | 설명 | 척도 |
|-----------|------|------|
| 리스크 | 적용 후 부작용 가능성 | 낮음/중간/높음 |
| 복잡도 | 구현 난이도 | 매우 낮음~매우 높음 |
| 영향 범위 | 변경이 미치는 범위 | 개인/조직/전체 |
| 소요 시간 | 적용까지 소요 시간 | 즉시/1주/1개월+ |
| 가역성 | 롤백 용이성 | 즉시 롤백/복잡/불가역 |
| 설계 부합성 | 원래 설계 의도와의 일치 | 부합/중립/상충 |

**산출물**: 최종 추천 (즉시 조치 + 중기 조치)

## 2. 보고서 템플릿

### 2.1 헤더 섹션

```markdown
# 전환배치 자동배치 최적화 Infeasible 원인 분석 및 해결방안 보고서 (N차)

| 항목 | 내용 |
|------|------|
| **문서번호** | HDTP-OPT-{YEAR}-{SEQ} |
| **작성일** | {DATE} |
| **작성자** | AI 분석팀 |
| **보안등급** | 대외비 |
| **DB 기준** | FTR_MOVE_STD_ID={ID}, CASE_ID={ID}, CASE_DET_ID='{ID}', REV_ID='{ID}' |
| **관련 문서** | 이전 차수 보고서 참조 |
```

### 2.2 섹션 구조

1. **개요** — 분석 배경, 이전 차수와의 비교
2. **문제 현상** — CPLEX 로그, IIS 요약
3. **데이터 현황** — 대상자, 조직, TO, 수급 분석
4. **근본 원인 — IIS 상세 분석**
   - IIS 구성 (TEAM040 N개 + 상대 제약 M개)
   - 충돌 구조도 (ASCII 다이어그램)
   - 수학적 증명
   - N차 분석 비교표
5. **충돌 제약 코드 분석** — 관련 제약 메서드 상세
6. **해결방안** — 총괄 비교표 + 각 방안 상세
7. **영향 범위** — 직접/간접 영향
8. **검증 방법** — 적용 후 확인 절차
9. **결론** — 패턴 요약, 즉시/중기 권고

### 2.3 포맷팅 규칙

- 충돌 구조도는 ASCII 박스 다이어그램 사용
- 수요 vs 공급 형태로 표현: `N명 수요 > M슬롯 공급 → 부족분 K`
- SQL은 항상 적용 SQL + 롤백 SQL 쌍으로 제공
- 모든 수치에 출처(LP 파일, DB, 코드) 명시
- 한국어 용어 사용 (하드 제약, 감점, 제약위배, 자동배치, 전환배치)

## 3. 역사적 패턴 데이터베이스

### 3.1 확인된 Infeasible 유형

| 유형 | 충돌 구조 | IIS 특징 | 해결 패턴 |
|------|-----------|---------|-----------|
| **개인 속성 충돌** | TEAMxxx(속성) vs TEAM040(직무) | IIS 2-3개, 1인 대상 | 속성 제약 CNST_GBN 변경 |
| **파트 TO 부족** | TEAM001(TO) vs TEAM040(직무) | IIS 10-20개, 복수 직원 | TO 증가 |
| **팀 용량 부족** | TEAM026(용량) vs TEAM040(직무) | IIS 20+개, 복수 직원 | 용량 증가 or 소프트 전환 |

### 3.2 공통 패턴

- **TEAM040은 항상 수요 측**: 모든 Infeasible 사례에서 TEAM040이 관여
- **부족분은 보통 1슬롯**: 대규모 부족이 아닌 경계 사례가 대부분
- **CNST_GBN 설정 오류**: 원래 소프트 의도였으나 하드로 설정된 경우 존재 (1차 TEAM030)
