## 1. 개요

전환배치계획 화면에서 자동배치(최적화)를 수행할 때, CPlex 솔버가 **"해가 존재하지 않음(Infeasible)"** 을 반환하는 문제가 발생하였다. 본 보고서는 해당 문제의 근본 원인을 분석하고 해결방안을 제시한다.

## 2. 문제 현상

### 2.1 CPlex 솔버 실행 결과

자동배치 최적화 수행 시 CPlex 솔버가 다음과 같은 Infeasibility 로그를 출력하였다.

```
Infeasibility row 'TEAM030.E784':  0 <= -1.
```

이는 **TEAM030** 제약이 직원 **E784**에 대해 수학적으로 만족 불가능한 상태임을 의미한다.

### 2.2 Conflict Refiner 결과

CPlex의 Conflict Refiner(충돌 분석기)를 실행한 결과, 전체 **19,856개** 제약 중 충돌의 원인이 되는 제약이 **2개**로 축소되었다.

| 충돌 제약 | 제약 코드 | 설명 |
|-----------|-----------|------|
| 충돌 제약 1 | **TEAM030** | 워킹맘 C지역(광역점) 배치금지 |
| 충돌 제약 2 | **TEAM040** | 가능직무내 이동 |

## 3. 시스템 구조

### 3.1 최적화 실행 흐름

자동배치 최적화는 다음과 같은 순서로 수행된다.

```
┌─────────────────────────────────────────────────────────────┐
│                    최적화 실행 파이프라인                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. MakeModel()                                             │
│     └─ 결정변수 생성: E{EMP_ID}.O{ORG_ID} (이진변수, 0/1)    │
│                                                             │
│  2. ApplyExcConstraint()     ─── 배타 제약 적용               │
│                                                             │
│  3. ApplyOrgConstraint()     ─── 조직 제약 적용               │
│                                                             │
│  4. ApplyTeamConstraint()    ─── 팀 제약 적용                 │
│                                                             │
│  5. ApplyPersonalConstraint()─── 개인 제약 적용               │
│     └─ TEAM030, TEAM040 포함                                 │
│                                                             │
│  6. MakeObjective()          ─── 목적식 구성 (감점 최소화)     │
│                                                             │
│  7. SolveWithOutCallback()   ─── CPlex 최적화 수행            │
│     ├─ 성공 → 배치 결과 반환                                  │
│     └─ 실패 → MakeConflictInfo() 호출                        │
│              └─ Conflict Refiner로 충돌 제약 분석              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 제약 유형 체계

본 시스템은 제약을 두 가지 유형으로 구분하며, DB의 `CNST_GBN` 컬럼 값에 따라 결정된다.

| 구분 | CNST_GBN 값 | IsPenaltyCnst | 수학적 처리 | 위반 가능 여부 |
|------|-------------|---------------|-------------|---------------|
| **하드 제약** | `"제약"` | `false` | 등식/부등식 직접 추가 | **절대 불가** |
| **소프트 제약** | `"감점"` | `true` | 슬랙 변수 + 목적식 패널티 | 가능 (감점 부여) |

- **하드 제약**: CPlex가 반드시 만족시켜야 하는 조건. 위반 시 해가 존재하지 않음(Infeasible).
- **소프트 제약**: 위반 시 목적식에 패널티(감점)가 부여되어 최적해 탐색 시 불이익을 주나, 해를 차단하지는 않음.

## 4. 충돌 제약 상세 분석

### 4.1 TEAM030: 워킹맘 C지역(광역점) 배치금지

#### DB 설정

| 필드 | 값 |
|------|-----|
| CNST_CD | TEAM030 |
| CNST_GBN | `"제약"` (하드 제약) |
| PENALTY_VAL | 10 |
| TAG1 | `"여성,기혼"` |
| USE_YN | 1 (사용) |

#### 코드 동작 (OptManager.cs:913-958)

1. `CNST_GBN="제약"` 이므로 `IsPenaltyCnst = false`
2. C지역 조직 목록 조회: `GetMegaBizOrg()` 메서드 사용
3. C지역 판별 기준: `MoveOrg.REGION_TYPE == "C"` (OrgModel.cs:319)
4. 제약식 생성:

```
AddLe( Sum(대상 직원의 C지역 변수), 0 )
```

> **의미**: 해당 직원을 C지역에 속한 **어떤 조직에도 배치할 수 없음** (합계가 0 이하).

#### 참고 사항

코드 주석(OptManager.cs:674)에는 이 제약이 **"배치지양"** 으로 기술되어 있어, 원래 소프트 제약(감점)으로 설계되었을 가능성이 높다. 또한 `PENALTY_VAL=10`이 DB에 이미 설정되어 있으나, `CNST_GBN="제약"`이므로 이 패널티 값은 코드에서 **완전히 무시**된다.

### 4.2 TEAM040: 가능직무내 이동

#### DB 설정

| 필드 | 값 |
|------|-----|
| CNST_CD | TEAM040 |
| CNST_GBN | `"제약"` (하드 제약) |
| PENALTY_VAL | None |
| USE_YN | 1 (사용) |

#### 코드 동작 (OptManager.cs:1301-1331)

1. 가능직무 조직 목록 조회: `GetJobTypeOrg()` 메서드 사용
2. 직무 매칭: `item.CaseItem.PROP9`에서 직무코드 추출 후 `MoveOrg.JOB_TYPE2`와 대조
3. 제약식 생성 (**IsPenaltyCnst 분기 없이 항상 하드 제약**):

```
AddGe( Sum(가능직무 조직 변수), 1 )
```

> **의미**: 가능직무에 해당하는 조직 중 **최소 1개에 반드시 배치**해야 함.

### 4.3 LP 파일 실증 분석

생성된 LP 파일(`emp_move_20260311_110244.lp`)에서 직원 E784에 대한 제약을 확인하였다.

#### TEAM030.E784 (LP 라인 73687)

```
E784.O12199 + E784.O12201 + ... + E784.O12249 <= 0
```

- E784의 **C지역 38개 조직 전체**에 대해 배치 금지
- 모든 이진변수의 합이 0 이하 → 어느 C지역 조직에도 배치 불가

#### TEAM040.E784 (LP 라인 133852)

```
_E784.O12235 >= 1
```

- **O12235에 반드시 배치** (가능직무 조직이 O12235 **단 1개**)
- 이진변수이므로 값이 반드시 1 → O12235에 배치 확정

#### 충돌 지점

| 사실 | 내용 |
|------|------|
| O12235의 REGION_TYPE | `"C"` (C지역, 광역점) |
| TEAM030 제약 | O12235 포함 C지역 38개 조직 배치 금지 |
| TEAM040 제약 | O12235에 반드시 배치 |

**O12235가 C지역에 속하므로 두 제약을 동시에 만족하는 것이 불가능하다.**

## 5. 근본 원인

### 5.1 직원 E784의 데이터 상태

```
┌──────────────────────────────────────────────────────┐
│                   직원 E784 프로필                      │
├──────────────────────────────────────────────────────┤
│  성별/혼인: 기혼여성(유자녀)  → TEAM030 대상            │
│  가능직무:  PROP9에 지정됨    → TEAM040 대상            │
│  가능직무 해당 조직: O12235   → 단 1개                  │
│  O12235 지역유형: C (광역점)                            │
└──────────────────────────────────────────────────────┘
```

### 5.2 논리적 충돌 구조

```
TEAM030 (하드 제약)          TEAM040 (하드 제약)
─────────────────           ─────────────────
"C지역 배치 금지"            "O12235에 반드시 배치"
Sum(C지역 변수) <= 0         Sum(가능직무 변수) >= 1
                 │                     │
                 │    O12235 ∈ C지역    │
                 │         ↓           │
                 └───── 충  돌 ─────────┘
                          │
                    ┌─────┴─────┐
                    │ INFEASIBLE │
                    └───────────┘
```

### 5.3 원인 요약

| 순서 | 원인 |
|------|------|
| 1 | 직원 E784는 기혼여성(유자녀)으로 TEAM030(C지역 배치금지) 대상 |
| 2 | 동시에 PROP9에 가능직무가 지정되어 TEAM040(가능직무내 이동) 대상 |
| 3 | E784의 가능직무에 해당하는 조직이 **O12235 단 1개**뿐 |
| 4 | O12235는 `REGION_TYPE="C"`로 **C지역(광역점)** 에 해당 |
| 5 | TEAM030은 `CNST_GBN="제약"`(하드)으로 C지역 배치를 **절대 금지** |
| 6 | TEAM040도 하드 제약으로 O12235에 **반드시 배치**를 요구 |
| 7 | **두 하드 제약의 동시 충족이 수학적으로 불가능** → Infeasible |

### 5.4 추가 발견

TEAM030의 DB 레코드에는 `PENALTY_VAL=10`이 설정되어 있다. 이 값은 소프트 제약(감점)에서만 사용되는 패널티 점수이며, 현재 `CNST_GBN="제약"`(하드 제약) 설정으로 인해 코드에서 **완전히 무시**되고 있다.

코드 주석(OptManager.cs:674)에도 해당 제약이 **"배치지양"** 으로 기술되어 있어, **원래 설계 의도는 소프트 제약(감점)이었을 가능성이 높다.**

### 6.2 방안 1: DB 데이터 변경 - TEAM030 CNST_GBN을 "감점"으로 변경 (즉시 조치 추천)

**변경 SQL:**

```sql
UPDATE MOVE_CASE_CNST_MASTER
   SET CNST_GBN = '감점'
 WHERE CNST_CD = 'TEAM030'
   AND FTR_MOVE_STD_ID = 202408
   AND CASE_ID = 2;

```

**변경 후 동작:**

- `IsPenaltyCnst = true`로 전환
- 제약식이 `AddLe(Sum, 0)` 에서 `AddEq(Sum, NewVar)` + 목적식에 `NewVar * 10` 추가로 변경
- CPlex는 C지역 배치 시 **10점 감점**을 부여하되 해를 찾을 수 있음

| 평가 항목 | 내용 |
|-----------|------|
| **장점** | 코드 수정 불필요, PENALTY_VAL=10 이미 존재, 즉시 적용 가능 |

| **리스크** | 중간 - 정책적 의사결정 필요 |
| **복잡도** | 매우 낮음 |
| **추천 근거** | 코드 주석 "배치지양" 및 PENALTY_VAL=10 기존재 → 원래 설계 의도에 부합 |

### 6.6 방안 5: 전처리 검증 - 충돌 제약 사전 검출 (중기 조치 추천)

자동배치 실행 전 전처리 단계에서, TEAM030 대상 직원과 TEAM040 대상 직원의 **교집합**을 검출하고, 해당 직원의 가능직무 조직이 **모두 C지역**인 경우 사전 경고를 발생시킨다.

**검증 로직:**

```
1. TEAM030 대상자 집합 A = { 기혼여성(유자녀) }
2. TEAM040 대상자 집합 B = { 가능직무 지정 직원 }
3. 교집합 C = A ∩ B
4. C의 각 직원에 대해:
   - 가능직무 조직 목록 조회
   - 모든 가능직무 조직이 C지역인 경우 → 경고 발생
```

| 평가 항목 | 내용 |
|-----------|------|
| **장점** | 문제를 사전에 경고, 다른 유사 충돌에도 확장 가능 |
| **단점** | 자동 해결이 아닌 경고만 제공, 구현 복잡도 높음 |
| **리스크** | 낮음 |
| **복잡도** | 높음 |

### 6.7 방안 6: 우선순위 체계 - 하드 제약 간 자동 완화

하드 제약에 우선순위를 부여하고, Infeasible 발생 시 낮은 우선순위의 제약을 자동으로 소프트 제약으로 완화하여 CPlex를 재호출하는 체계를 구축한다.

| 평가 항목 | 내용 |
|-----------|------|
| **장점** | 범용적 해결 가능 |

| **리스크** | 높음 |
| **복잡도** | 매우 높음 |

## 7. 권고사항

### 7.1 즉시 조치 (방안 1)

> **TEAM030의 CNST_GBN을 "제약"에서 "감점"으로 변경한다.**

**근거:**

1. 코드 주석(OptManager.cs:674)에 **"배치지양"** 으로 기술되어 있어 원래 소프트 제약으로 설계된 것으로 판단됨
2. `PENALTY_VAL=10`이 이미 DB에 설정되어 있어 감점 전환 시 별도 값 지정 불필요
3. 코드 수정 없이 DB 변경만으로 즉시 적용 가능
4. C지역 배치 시 10점 감점이 부여되므로 최적화 과정에서 자연스럽게 C지역 배치를 회피함

### 7.1.1 롤백 SQL

문제 발생 시 다음 SQL로 원래 상태로 복구한다.

```sql
-- 롤백 SQL (문제 발생 시)
UPDATE MOVE_CASE_CNST_MASTER
SET CNST_GBN = '제약'
WHERE CNST_CD = 'TEAM030'
  AND FTR_MOVE_STD_ID = 202408
  AND CASE_ID = 2;
COMMIT;
```

### 7.2 중기 조치 (방안 5)

> **전처리 단계에 제약 간 충돌 사전 검출 로직을 구현한다.**

**근거:**

1. 유사한 제약 충돌이 다른 직원이나 다른 케이스에서도 발생할 수 있음
2. Infeasible 발생 후 분석하는 것보다 사전 방지가 효율적
3. 사용자에게 명확한 경고 메시지를 제공하여 데이터 수정 기회를 부여

## 10. 결론

전환배치 자동배치 시 발생한 Infeasible 문제는 **TEAM030(워킹맘 C지역 배치금지)** 과 **TEAM040(가능직무내 이동)** 두 하드 제약이 직원 E784에 대해 **수학적으로 동시 충족 불가능한 상태**를 형성한 것이 원인이다.

직원 E784는 기혼여성으로 TEAM030에 의해 C지역 배치가 금지되나, 가능직무 조직이 C지역의 O12235 단 1개뿐이어서 TEAM040에 의해 해당 조직에 반드시 배치되어야 하는 모순이 발생한다.

**TEAM030의 DB 설정(`CNST_GBN`)을 "제약"에서 "감점"으로 변경**하는 것을 즉시 조치로 권고한다. 이는 코드 주석의 "배치지양" 기술 및 `PENALTY_VAL=10` 기설정 사실에 비추어 원래 설계 의도에 부합하며, 코드 수정 없이 DB 변경만으로 즉시 해결 가능하다.

중기적으로는 **전처리 충돌 검출 로직**을 구현하여 유사한 제약 충돌을 사전에 방지하는 체계를 구축할 것을 권고한다.
