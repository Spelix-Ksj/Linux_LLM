## 1. 보고서 개요

### 1.1 목적

본 보고서는 현재 운영 중인 Local vLLM 서버(gpt-oss-120B)가 Claude Opus 4.6이 수행한 **"전환배치 자동배치 Infeasible 분석 보고서"**(1차/2차)와 동일한 수준의 분석을 수행할 수 있는지를 평가하고, 가능하다면 어떤 조건과 전략이 필요한지를 제시하는 것을 목적으로 한다.

### 1.2 배경

2026년 3월, Claude Opus 4.6 기반 Claude Code 환경에서 전환배치 자동배치 시스템의 Infeasible(풀이 불가) 원인을 분석하는 보고서를 작성하였다. 이 분석은 다음을 포함하였다:

- **22MB LP 파일** 파싱 및 제약식 추출
- **OptManager.cs(3,745줄)** 등 대규모 C# 소스코드 전체 이해
- **Oracle DB 10개 쿼리** 실행 및 교차 분석
- **CPlex Conflict Refiner / IIS** 결과 해석
- **10개 장 구조의 체계적 보고서** 자동 생성

본 보고서는 이 분석 과정에서 요구된 역량을 체계적으로 분류하고, Local LLM의 현재 역량과의 갭을 정량/정성적으로 평가한다.

### 1.3 핵심 질문

> **현재 운영 중인 Local vLLM(gpt-oss-120B) 서버로 Claude Opus 4.6이 수행한 수준의 "전환배치 자동배치 Infeasible 분석 보고서"와 동일한 분석이 가능한가? 가능하려면 어떻게 해야 하는가?**

## 3. Claude Opus 4.6이 수행한 분석의 요구 역량

1차/2차 Infeasible 분석 보고서 작성 과정에서 사용된 역량을 6개 범주로 분류한다.

### 3.1 [A] 다단계 추론 (Multi-step Reasoning)

Claude Opus 4.6은 다음과 같은 **10단계 이상의 추론 체인**을 수행하였다:

```
[추론 체인 예시]

1단계: OptManager.cs에서 TEAM030_PENALTY() 메서드 진입
   ↓
2단계: IsPenaltyCnst 분기 로직 추적
   ↓
3단계: CNST_GBN 값에 따른 분기 확인
   ↓
4단계: IsPenaltyCnst = true → AddLe (부등식, 소프트 제약)
        IsPenaltyCnst = false → AddEq (등식, 하드 제약)
   ↓
5단계: DB에서 해당 제약의 CNST_GBN 값 조회
   ↓
6단계: CNST_GBN = "제약" → IsPenaltyCnst = false 확인
   ↓
7단계: 하드 제약(AddEq)으로 처리됨을 확인
   ↓
8단계: 코드 주석 "배치지양"과 DB 설정 "제약" 간 불일치 발견
   ↓
9단계: 이 불일치가 Infeasible의 근본 원인임을 추론
   ↓
10단계: 해결방안 (CNST_GBN을 "배치지양"으로 변경) 도출
```

**핵심**: C# 소스코드 3,745줄 전체를 이해한 상태에서, 메서드 호출 관계와 조건 분기를 추적하고, DB 설정값과 교차 검증하여 **인과관계를 도출**하는 능력.

### 3.2 [B] 대규모 컨텍스트 처리

| 처리 대상 | 규모 | 추정 토큰 수 |
|-----------|------|-------------|
| OptManager.cs | 3,745줄 | ~15,000 토큰 |
| TransferPlanManager.cs | 3,742줄 | ~15,000 토큰 |
| ConstraintModel.cs | 2,273줄 | ~9,000 토큰 |
| LP 파일 (22MB) | 수십만 줄 | ~수백만 토큰 (선별 추출) |
| DB 쿼리 결과 10건 | 다양 | ~5,000 토큰 |
| 분석 보고서 출력 | 10개 장 | ~10,000 토큰 |
| **총 처리 컨텍스트** | | **약 10만 토큰 이상** |

**핵심**: 200K 토큰 컨텍스트 윈도우를 활용하여 다수의 대규모 파일을 동시에 보유하고 교차 참조하는 능력.

### 3.3 [C] 도구 사용 (Tool Use / Agentic Workflow)

분석 과정에서 활용된 도구 목록:

| 도구 | 용도 | 사용 빈도 |
|------|------|----------|
| **Oracle DB 접속** (Bash + sqlplus) | 제약조건, TO, 직원정보 등 조회 | 10회+ |
| **파일 시스템 탐색** (Glob) | 소스코드 파일 위치 탐색 | 5회+ |
| **코드 내용 검색** (Grep) | 특정 메서드, 변수명 검색 | 15회+ |
| **파일 읽기** (Read) | C# 소스코드, LP 파일 읽기 | 20회+ |
| **코드 심볼릭 분석** (Serena MCP) | 메서드 호출 관계, 심볼 정의 추적 | 5회+ |
| **병렬 에이전트** (TaskCreate) | 3-4개 에이전트 동시 분석 수행 | 3-4회 |
| **파일 쓰기** (Write) | 최종 보고서 생성 | 2회 |

**핵심**: 단일 LLM 호출이 아닌, **자율적으로 도구를 선택하고 조합하는 에이전트**로서 동작. 특히 병렬 에이전트 운영을 통해 코드 분석, DB 분석, LP 분석을 동시에 수행.

### 3.4 [D] 도메인 지식 통합

| 도메인 | 필요 지식 | 활용 장면 |
|--------|----------|----------|
| **MILP** (Mixed Integer Linear Programming) | 선형계획법, 정수변수, 목적함수, 제약식 | LP 파일 해석, Infeasible 원인 분석 |
| **CPlex 솔버** | Conflict Refiner, IIS(Irreducible Infeasible Set), 하드/소프트 제약 | IIS 결과 해석, 제약 완화 전략 수립 |
| **LP 파일 문법** | 변수명 규칙, 제약식 표현, RHS, Bounds, Generals | 22MB LP 파일에서 관련 제약 추출 |
| **인사 배치 최적화** | TO(Target Of), 잔류원, 가능직무, 배치지양/불가 | 비즈니스 로직 이해 및 보고서 작성 |
| **C# / .NET** | 클래스 구조, LINQ, 조건문 분기 | OptManager.cs 소스코드 분석 |
| **Oracle SQL** | PL/SQL, 딕셔너리 뷰, JOIN, 서브쿼리 | DB 조회 쿼리 작성 및 실행 |

**핵심**: 사전학습에서 MILP, CPlex, LP 파일 형식, 인사관리 도메인에 대한 포괄적 지식을 보유하고 있으며, 실시간 코드/DB 탐색으로 부족한 부분을 보완.

### 3.5 [E] 교차 분석 (Cross-referencing)

분석에서 수행한 **4개 소스 간 교차 검증**:

```
    [C# 소스코드]          [Oracle DB]
         │                      │
         │  CNST_GBN 값 매칭     │
         ├──────────────────────┤
         │                      │
         │  제약식 ID 매칭       │
         ├──────────────────────┤
         │                      │
    [LP 파일]             [비즈니스 로직]
         │                      │
         │  변수명↔직원ID 매칭   │
         ├──────────────────────┤
         │  제약식↔TO 매칭       │
         └──────────────────────┘
```

구체적 교차 분석 사례:

| 교차 분석 | 상세 |
|-----------|------|
| **코드 ↔ DB** | TEAM030_PENALTY()의 IsPenaltyCnst 분기 ↔ DB의 CNST_GBN 값 |
| **LP ↔ DB** | TEAM040 제약식의 대상 변수 ↔ DB의 직원 9명 식별 |
| **LP ↔ 코드** | LP 파일의 제약명 패턴 ↔ C# 코드의 제약 생성 로직 |
| **DB ↔ 비즈니스** | TO 값 = 0인 조직 6개 ↔ 배치 불가능 조직 식별 |
| **IIS ↔ 전체** | IIS 15개 제약 ↔ 코드+DB+LP+비즈니스 로직 종합 매칭 |

**핵심**: 단일 소스 분석이 아닌, **이질적인 4개 소스를 동시에 참조하며 일관성을 검증**하는 능력.

## 4. 갭 분석

### 4.1 역량별 갭 분석 총괄표

| 역량 | Local LLM 현재 수준 | 필요 수준 | 갭 크기 | 갭 등급 |
|------|---------------------|----------|---------|---------|
| **A. 다단계 추론** | 3-4단계 (복합 SQL) | 10+단계 (인과추론) | **매우 큼** | Critical |
| **B. 대규모 컨텍스트** | 8K-32K 토큰 | 100K+ 토큰 | **매우 큼** | Critical |
| **C. 도구 사용** | Text2SQL 단일 파이프라인 | 멀티도구 에이전트 | **매우 큼** | Critical |
| **D. 도메인 지식** | Text2SQL 한정 프롬프트 | MILP+CPlex+LP+인사 | **큼** | Major |
| **E. 교차 분석** | 단일 소스(DB) | 4개 소스 동시 교차 | **매우 큼** | Critical |
| **F. 보고서 구조화** | SQL 결과 요약 수준 | 10장 공식 보고서 | **큼** | Major |

> **Critical 4건, Major 2건**: 6개 역량 중 4개가 "매우 큼(Critical)" 수준의 갭을 보인다.

### 4.2 상세 갭 분석

#### [A] 다단계 추론 갭

```
현재 수준                          필요 수준
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
│ 자연어 → SQL 변환              │ C# 3,745줄 전체 이해        │
│ 3-4단계 추론 체인              │ 10+단계 추론 체인           │
│ 복합 쿼리 30-40% 정확도        │ 인과관계 도출 필요           │
│                               │ 코드↔DB↔LP 연결 추론        │
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
        ←────── 갭: Critical ──────→
```

- **gpt-oss-120b**: MoE 구조로 117B 파라미터이나, 활성 파라미터는 이보다 적다. Text2SQL에서 복합 쿼리 30-40% 정확도는 다단계 추론 능력의 명확한 한계를 보여준다.
- **Claude Opus 4.6**: 수백만 토큰 컨텍스트에서 10단계 이상의 추론 체인을 수행하며, 코드 분기 로직→DB 설정값→LP 제약식→비즈니스 의미까지의 인과관계를 도출한다.
- **갭 분석**: 추론 깊이의 차이는 모델 아키텍처와 학습 데이터 규모에서 기인하며, 단순 프롬프트 엔지니어링으로 해소하기 어렵다.

#### [B] 대규모 컨텍스트 갭

```
현재 수준                          필요 수준
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
│ 8K-32K 토큰                   │ 100K+ 토큰                 │
│ (EXAONE 8K 초과 이력)          │ (200K 컨텍스트 윈도우)       │
│ 단일 파일 부분 처리             │ 다수 파일 동시 보유          │
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
        ←────── 갭: Critical ──────→
```

- **gpt-oss-120b**: vLLM 설정에 따라 8K-32K 수준. EXAONE에서 8K 초과 문제가 발생하여 16K로 확장한 이력이 있으며, 이는 컨텍스트 제한이 실질적 병목임을 시사한다.

## 5. 결론: 현재 수준으로 동일 분석이 가능한가

### 5.1 최종 결론

> **현재 상태의 Local LLM(gpt-oss-120b)으로 Claude Opus 4.6이 수행한 수준의 Infeasible 분석은 불가능하다.**

### 5.2 불가능 판정 근거

#### 근거 1: 모델 역량의 근본적 차이

gpt-oss-120b는 117B MoE 파라미터이나, MoE(Mixture of Experts) 아키텍처 특성상 **실제 활성 파라미터는 이보다 상당히 적다**. Claude Opus 4.6 대비 추론 깊이, 컨텍스트 길이, 도메인 지식 범위에서 근본적 차이가 존재한다.

이를 뒷받침하는 정량적 지표:

| 지표 | gpt-oss-120b | Claude Opus 4.6 | 비율 |
|------|-------------|-----------------|------|
| 복합 쿼리 정확도 | 30-40% | 85%+ | 약 1:2.5 |
| 최대 추론 단계 | 3-4단계 | 10+단계 | 약 1:3 |
| 컨텍스트 윈도우 | 8K-32K | 200K | 약 1:6~25 |

Text2SQL에서 복합 쿼리 30-40% 정확도는 **다단계 추론 능력의 상한선**을 보여주는 지표이다. Infeasible 분석은 이보다 훨씬 복잡한 10단계 이상의 인과관계 추론을 요구한다.

#### 근거 2: 시스템 아키텍처의 부재

방금 수행한 분석은 LLM이 "도구 사용 에이전트(Tool-using Agent)"로서 동작한 결과이다:

- Oracle DB에 직접 접속하여 쿼리를 실행
- 파일 시스템을 탐색하여 소스코드를 찾고 읽기
- 코드 심볼릭 분석 도구로 메서드 호출 관계를 추적
- 3-4개 병렬 에이전트를 운영하여 동시 분석을 수행

현재 Local LLM은 **Text2SQL 단일 파이프라인만 구현**되어 있다. "자연어 → SQL → 결과"의 단방향 흐름만 지원하며, 파일 읽기, 코드 분석, 병렬 작업 등의 에이전틱 워크플로우가 불가능하다.

#### 근거 3: 컨텍스트 윈도우의 물리적 한계

Infeasible 분석에 필요한 동시 처리 데이터 규모:

| 데이터 | 크기 | 현재 컨텍스트 대비 |
|--------|------|------------------|
| OptManager.cs (3,745줄) | ~15K 토큰 | 32K의 47% |
| TransferPlanManager.cs (3,742줄) | ~15K 토큰 | 32K의 47% |
| ConstraintModel.cs (2,273줄) | ~9K 토큰 | 32K의 28% |
| DB 쿼리 결과 10건 | ~5K 토큰 | 32K의 16% |
| LP 파일 관련 제약 발췌 | ~10K 토큰 | 32K의 31% |
| **합계** | **~54K 토큰** | **32K의 169%** |

**최대 컨텍스트 32K를 가정하더라도 물리적으로 필요 데이터의 59%만 적재 가능하다.** 8K인 경우 약 15%만 적재 가능하여, 교차 분석 자체가 성립하지 않는다.

## 6. 동일 수준에 도달하기 위한 학습/구축 전략

### 6.1 단기 전략 (1-3개월): Agentic RAG 시스템 구축

**목표: 현재 대비 50% 수준 도달**

| 항목 | 내용 | 예상 효과 |
|------|------|----------|
| **에이전트 프레임워크** | LangGraph 또는 CrewAI 기반 멀티스텝 에이전트 구축 | 도구 사용 역량 확보 |
| **도구 통합** | Oracle DB 접속, 파일 읽기/검색, 코드 분석 도구를 LLM에 연결 | 교차 분석 기초 확보 |
| **RAG 구축** | C# 소스코드, LP 파일, 제약조건 문서를 벡터 DB에 색인 | 컨텍스트 한계 부분 보완 |
| **컨텍스트 확장** | vLLM max_model_len을 모델 지원 최대치로 확장 | 동시 처리량 증가 |
| **도메인 프롬프트** | MILP, CPlex, LP 파일 해석 관련 도메인 지식을 SYSTEM_PROMPT에 추가 | 도메인 지식 보완 |
| **CoT 프롬프트** | 분석 단계를 명시적으로 분해하는 Chain-of-Thought 프롬프트 설계 | 추론 깊이 증가 |

**단기 전략 아키텍처:**

```
[사용자 질의]
     │
     ▼
[오케스트레이터 에이전트] ── gpt-oss-120b
     │
     ├── [도구 1] Oracle DB 접속 + SQL 실행
     │
     ├── [도구 2] 파일 시스템 읽기/검색 (RAG)
     │
     ├── [도구 3] 코드 분석 (AST 파싱)
     │
     └── [도구 4] LP 파일 파싱 (전용 파서)
            │
            ▼
     [결과 종합 + 보고서 생성]
```

**한계**: 에이전트 단계 간 정보 전달의 정확도가 낮을 수 있으며, RAG의 검색 품질이 분석 품질을 좌우한다.

## 7. 현실적 권고

### 7.1 가장 빠르고 효과적인 방법

**권고안 1: 하이브리드 아키텍처 (최우선 권고)**

```
[사용자 질의]
     │
     ▼
[Local LLM: 1차 분류 + 데이터 수집]
  ├── 문제 유형 분류 (Infeasible / 성능 / 데이터)
  ├── 관련 DB 데이터 사전 수집
  ├── 관련 소스코드 영역 식별
  └── LP 파일에서 관련 제약 추출
     │
     ▼
[Claude API: 복합 분석 + 보고서 생성]
  ├── 수집된 데이터 기반 다단계 추론
  ├── 교차 분석 수행
  └── 체계적 보고서 생성
```

- **Local LLM** (gpt-oss-120b): Text2SQL, 데이터 수집, 문제 분류 등 **경량 작업 담당**
- **Claude API**: 복잡한 분석, 다단계 추론, 교차 분석, 보고서 생성 등 **고난도 작업 담당**
- **이점**: 즉시 구현 가능, Local LLM의 기존 역량 활용, API 비용 최적화

**권고안 2: Local LLM을 1차 분류기로 활용**

| 단계 | 담당 | 내용 |
|------|------|------|
| 1단계 | Local LLM | 문제 유형 분류 (Infeasible / 성능 / 데이터 오류 등) |
| 2단계 | Local LLM | 관련 데이터 수집 (DB 쿼리, 파일 검색) |
| 3단계 | Local LLM | 기본 분석 수행 (패턴 매칭, 규칙 기반) |
| 4단계 | 상위 모델 | 심층 분석 및 인과관계 추론 |
| 5단계 | 상위 모델 | 교차 검증 및 보고서 생성 |

**권고안 3: 분석 지식 축적 (RAG)**

- 매번 수행한 분석 결과를 벡터 DB에 저장
- 유사 문제 발생 시 기존 분석을 참조하는 RAG 방식
- 시간이 지남에 따라 Local LLM만으로도 처리 가능한 범위가 확대

### 7.2 현실적 평가

> **핵심 판단: 120B MoE 모델로 Claude Opus 4.6 수준의 분석을 단독으로 재현하는 것은, 파인튜닝만으로는 달성하기 극히 어렵다.**

그 이유는 다음과 같다:

1. **모델 크기/아키텍처의 근본적 차이**: MoE 아키텍처에서 활성 파라미터 수와 Dense 모델의 전체 파라미터 수 사이에는 추론 깊이에 직접적으로 영향하는 차이가 있다. 이는 학습 데이터나 프롬프트로 완전히 보상할 수 없는 구조적 한계이다.

2. **컨텍스트 윈도우의 물리적 한계**: 8K-32K 토큰으로 100K+ 토큰이 필요한 분석을 수행하려면, 정보를 분할하고 단계별로 처리해야 한다. 이 과정에서 정보 손실이 불가피하며, 교차 분석의 품질이 저하된다.

3. **에이전트 오케스트레이션의 복잡도**: "작은 단계의 문제로 분해하여 120B 모델이 각 단계를 처리"하는 전략이 가장 현실적이나, 단계 간 정보 전달의 정확성이 전체 분석 품질을 좌우하며, 오케스트레이션 자체의 설계와 유지보수 비용이 높다.

### 7.3 비용-효과 분석

| 전략 | 구현 비용 | 기간 | 달성 수준 | 유지보수 | 권고 |
|------|----------|------|----------|---------|------|
| 하이브리드 (Local + Claude API) | 낮음 | 1-2주 | 90%+ | 낮음 | **최우선** |
| Agentic RAG 구축 | 중간 | 1-3개월 | 50% | 중간 | 병행 추진 |
| LoRA 파인튜닝 | 높음 | 3-6개월 | 70% | 높음 | 선택적 |
| 전문 에이전트 시스템 | 매우 높음 | 6개월+ | 85% | 매우 높음 | 장기 과제 |

## 8. 모델 크기별 역량 비교

### 8.1 종합 비교표

| 역량 | 7B | 30B (Qwen3-Coder) | 120B MoE (gpt-oss) | Claude Opus 4.6 |
|------|:---:|:---:|:---:|:---:|
| 단순 Text2SQL | 60-65% | 70-75% | 55-60% (Oracle) | 90%+ |
| 복합 쿼리 | 20-30% | 40-50% | 30-40% (Oracle) | 85%+ |
| 다단계 추론 | 제한적 | 보통 | 보통~양호 | 우수 |
| 코드 이해 (3,000줄+) | 불가 | 제한적 | 가능 (분할 시) | 우수 |
| LP 파일 해석 | 불가 | 제한적 | 가능 (프롬프트) | 우수 |
| 교차 분석 (4개 소스) | 불가 | 제한적 | 에이전트로 가능 | 네이티브 |
| 보고서 구조화 | 기본 | 양호 | 양호 | 우수 |
| 컨텍스트 윈도우 | 4K-8K | 32K-128K | 8K-32K | 200K |
| 도구 사용 (Agentic) | 제한적 | 보통 | 양호 | 우수 |

### 8.2 핵심 시사점

- **gpt-oss-120b의 Oracle Text2SQL 정확도(55-60%)가 30B 모델(70-75%)보다 낮은 이유**: Oracle 전용 학습 데이터 부족, MoE 활성 파라미터 효율, 프롬프트 최적화 미흡 등이 복합적으로 작용한 것으로 추정된다.
- **Qwen3-Coder-30B의 활용 가능성**: 코드 분석 전문 에이전트로 활용 시, C# 소스코드 분석 역할을 담당할 수 있다. 다만 단독으로 전체 분석을 수행하기에는 컨텍스트와 추론 깊이가 부족하다.
- **Claude Opus 4.6의 차별점**: 단일 모델이 모든 역량(추론+컨텍스트+도구+도메인+교차+보고서)을 높은 수준으로 통합 보유하는 것이 핵심 강점이다. Local LLM으로 이를 재현하려면 여러 시스템의 조합이 필요하다.

## 9. 부록: 용어 정의

| 용어 | 정의 |
|------|------|
| **MoE** (Mixture of Experts) | 입력에 따라 일부 전문가(Expert) 네트워크만 활성화하는 아키텍처. 총 파라미터 대비 실제 연산에 사용되는 활성 파라미터가 적다. |
| **vLLM** | 대규모 언어모델 서빙을 위한 고성능 추론 엔진. PagedAttention 등 메모리 최적화 기술을 적용. |
| **Tensor Parallel** | 모델의 텐서를 여러 GPU에 분산하여 병렬 처리하는 기법. |
| **MILP** (Mixed Integer Linear Programming) | 일부 변수가 정수인 선형계획법. 전환배치 최적화에 사용. |
| **CPlex** | IBM의 수리최적화 솔버. MILP 문제 풀이에 사용. |
| **IIS** (Irreducible Infeasible Set) | 풀이 불가능한 문제에서 최소한의 충돌 제약 집합. |
| **LP 파일** | CPlex 입력 형식의 선형계획 문제 기술 파일. |
| **Infeasible** | 주어진 모든 제약을 동시에 만족하는 해가 존재하지 않는 상태. |
| **RAG** (Retrieval-Augmented Generation) | 외부 문서를 검색하여 LLM의 컨텍스트에 추가하는 기법. |
| **LoRA** (Low-Rank Adaptation) | 소수의 파라미터만 추가 학습하는 효율적 파인튜닝 기법. |
| **CoT** (Chain-of-Thought) | LLM이 단계별로 사고 과정을 명시하도록 유도하는 프롬프트 기법. |
| **LangGraph** | LangChain 기반의 상태 머신 에이전트 프레임워크. |
| **CNST_GBN** | 제약 구분 코드. "제약"=하드 제약, "배치지양"=소프트 제약. |
| **TO** (Target Of) | 목표 배치 인원. 특정 조직에 배치할 목표 인원 수. |
